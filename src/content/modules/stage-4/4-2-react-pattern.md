## 为什么学这个

Agent 并不是“想到什么就做什么”的黑箱。它之所以能在复杂任务里连续推进，背后通常依赖一种可重复的决策模式。ReAct 是最重要的入门框架之一，因为它把 Agent 的内部过程拆成了清楚的循环: 先思考当前状态，再决定采取什么行动，再根据行动结果继续修正。没有这层结构，很多所谓 Agent 实际上只是把若干工具粗暴拼在一起。

学习 ReAct 的价值，不只是会写一种 prompt，更是学会分析 Agent 为什么会用某个工具、为什么会卡在某一步、为什么会反复重复错误。当你把 ReAct 视为“可观察的决策骨架”，后面不论是看 LangChain Agent、自己写简单循环，还是排查工具误用，都更容易定位问题。

## 核心知识

ReAct 的核心思想可以概括为三个阶段反复循环: Thought、Action、Observation。Thought 负责当前局面的判断，比如信息是否足够、下一步目标是什么、是否需要额外证据；Action 负责把决策落成一个明确动作，例如调用搜索工具、运行计算器、查询数据库；Observation 则是行动结果返回给模型后的新输入，帮助模型判断是该继续前进、换工具，还是已经可以直接回答。这种循环让模型不必在一开始就把整条路径想完，而是允许“边做边看”。

在 Prompt 结构上，ReAct 不是单纯列几个英文标签，而是通过格式约束把中间步骤外显出来。一个典型模板会要求模型先写当前判断，再输出某个工具名与参数，接着读取外部系统返回的 Observation，再决定下一轮。这里的重要性在于把“推理”和“执行”拆开。你不希望模型在 Thought 里伪造工具结果，也不希望它在还没足够信息时强行作答。结构化循环能减少这种混杂。

工具选择决策是 ReAct 的关键能力之一。不是每一步都应该调用工具。若问题属于通用知识、无需实时数据、也不依赖私有环境，模型直接回答通常更快更便宜；若问题需要最新信息、精确计算、访问内部数据或跨步骤执行，工具调用才有价值。一个成熟的 Agent 不是“能用工具就用”，而是能评估工具成本、失败概率和必要性。很多初学者的第一个误区，就是让模型把简单问题也送进搜索，最后得到更慢、更贵、还更不稳定的回答。

ReAct 也有明显局限。它本质上是线性推进的，一步接一步，缺少全局回溯能力。如果前几步路线选错，后面常常在错误路径上越走越远。它还容易陷入循环，例如不断搜索同义词、重复调用同一个失败工具，或者在“还差一点信息”的幻觉中迟迟不收敛。再往深一点看，ReAct 暴露出的中间思路虽然便于调试，但在生产中也可能带来安全、成本和稳定性问题，因此很多框架会把内部 reasoning 与用户可见输出分离。

## 关键对比 / 易错点

- ReAct 的重点不是“写出 Thought 字样”，而是让模型在行动前显式区分判断和执行。
- 不是所有问题都值得进入 think-act-observe 循环。直接回答有时反而更稳、更便宜。
- Observation 必须来自真实工具结果，而不是模型自己脑补的返回值，否则整个循环会建立在假数据上。
- ReAct 擅长逐步探索，但不擅长复杂回溯和并行规划；任务一长，容易线性失真。
- 看到 Agent 连续调用很多步，不代表它在认真工作，也可能是在无效循环。

## 学习检查点

- 你能否分别解释 Thought、Action、Observation 在 ReAct 里各自负责什么？
- 遇到一个无需实时信息的常识问题时，为什么“直接回答”可能比“先搜索再回答”更合理？
- 如果工具已经返回空结果，Agent 还在重复同类搜索，这暴露了 ReAct 的什么局限？
- 为什么说 ReAct 既是一种 prompt 技术，也是一种排障视角？

## 实践建议

- 手写一个最小 ReAct 示例，哪怕只包含“是否需要搜索”这一个工具决策，也要把 Thought、Action、Observation 明确分开，训练对循环的感觉。
- 在你后续实现 Agent 时，先记录每一步的工具选择理由和 Observation，再看最终答案；这样一旦结果异常，你能更快判断是决策问题还是工具问题。
