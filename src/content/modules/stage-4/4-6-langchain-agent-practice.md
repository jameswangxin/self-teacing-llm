## 为什么学这个

前面几课学到的是 Agent 的组件和机制，但如果不能把它们串成一个可运行、可观察、可排障的系统，知识就很容易停留在概念层。LangChain 的价值，在于它把工具、模型、记忆和执行器组织成相对统一的开发接口，让你能更快从“理解原理”过渡到“搭出第一个 Agent 实战项目”。

但也正因为 LangChain 帮你封装了很多细节，初学者很容易产生另一种错觉: 只要 `create_agent` 或 `AgentExecutor` 跑起来，Agent 就算完成了。真实工程里，决定系统能不能用的往往是可观测性、调试能力和故障排查，而不是 Demo 首次成功。这个模块的目标，就是把你带到“能构建，也能解释和修复”的阶段。

## 核心知识

LangChain 构建工具型 Agent 的常见流程可以概括为: 定义模型、注册工具、准备 prompt 或 agent 类型、创建 agent、再由 `AgentExecutor` 负责循环执行。`AgentExecutor` 的职责不是简单调用一次模型，而是接管整个 think-act-observe 循环: 把用户输入和上下文送入 Agent、接收模型提出的工具调用、执行工具、把结果作为 observation 回填，再判断是否进入下一轮或输出最终答案。理解这个执行器角色非常重要，因为很多“Agent 出问题”的根源，其实发生在循环控制和中间状态，而不是模型本身。

Verbose 日志是最便宜也最常用的第一层调试手段。打开 verbose 后，你通常能看到每一步的输入、工具选择、工具返回和最终输出，这有助于快速判断 Agent 是“不会选工具”“选对了但参数错”“工具成功但总结错”，还是根本没有进入预期路径。LangSmith 则把这种观察能力进一步结构化成 trace，你可以回看每一轮调用树、链路耗时、输入输出和失败节点。它的价值不只是看热闹，而是帮你把偶发失败变成可复盘的证据。

死循环、工具误用和推理偏差，是实践里最常见的三类问题。死循环通常表现为重复调用同一工具或在多个近似查询之间来回跳；工具误用表现为模型拿搜索去做数学题、拿数据库工具查不存在的字段；推理偏差则是模型虽然拿到了正确 observation，却得出了错误结论。对应的排查顺序应是先看 trace 中的决策链，再判断是 prompt 约束不清、工具描述有缺口、执行器轮次限制太宽，还是 observation 格式让模型读不懂。

研究助理 Agent 是一个很适合 Stage 4 的实战项目，因为它天然覆盖多步搜索、文档阅读、证据整合和报告输出。一个最小版本不需要追求“自动完成整个科研流程”，只要能围绕一个主题完成检索、去重整理、提炼要点和生成结构化摘要，就已经足够练到 Agent Executor、工具组合、可观测性和排障的关键肌肉。项目范围控制得越清楚，你越容易把调试精力放在核心机制上，而不是被无穷扩展需求拖走。

## 关键对比 / 易错点

- `AgentExecutor` 不只是“包装器”，它实际负责多轮循环控制，是 Agent 行为的重要一层。
- Demo 能跑通不等于系统可维护；没有 verbose 或 trace，很多问题只能靠猜。
- Agent 死循环时，先缩轮次和看轨迹，往往比继续加工具更有效。
- 工具误用不一定是模型太笨，很多时候是工具描述、返回格式或停止条件设计得不好。
- 研究助理项目的重点是练闭环，不是把所有联网、写作、引用能力一次做满。

## 学习检查点

- `AgentExecutor` 在 LangChain Agent 流程里具体负责哪些事情？
- 打开 verbose 日志后，你最希望先观察哪几类中间信息？为什么？
- LangSmith trace 相比普通控制台日志，额外解决了什么问题？
- 如果 Agent 一直重复搜索相近关键词，你会优先从哪些地方排查？

## 实践建议

- 实现一个最小研究助理 Agent 时，先把工具限制在 2 到 3 个，并设置明确的最大轮次，先把循环稳定下来再扩能力。
- 每跑一次调试案例都保存 trace 或关键日志，手工总结“本次失败是决策问题、工具问题还是 observation 问题”，为后续优化建立自己的故障样本库。
