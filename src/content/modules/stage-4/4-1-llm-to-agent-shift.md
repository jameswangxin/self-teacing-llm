## 为什么学这个

很多人第一次接触 Agent 时，会把它理解成“更会说话的 LLM”或者“带插件的聊天机器人”。这个理解不够准确。纯 LLM 对话的默认模式，是你给一次输入，它返回一次输出；任务边界、是否继续、是否查资料、是否保存状态，通常都由人来决定。Agent 的变化在于，系统开始能够围绕目标持续执行一个闭环，而不是停留在单轮文本生成。

这一课的价值，是帮你把“模型能力”升级为“系统能力”的视角。你要开始用组件分工而不是单个 prompt 来理解 AI 应用: 哪部分负责推理，哪部分负责行动，哪部分负责记忆，哪部分负责拆解任务。只有把这层范式跃迁看清楚，后面学习 ReAct、Function Calling、工具和记忆时，才不会把所有问题都误判成 prompt 问题。

## 核心知识

纯 LLM 对话和 Agent 的本质区别，不在于模型是不是更聪明，而在于是否形成了“感知环境 -> 做决策 -> 采取行动 -> 根据结果继续调整”的执行闭环。普通聊天更像你每次都在手动驾驶: 你决定下一步该问什么、该查什么、该补什么上下文。Agent 则更像半自动或自动驾驶系统: 它在目标约束内自主决定要不要用工具、要不要继续搜索、要不要把中间结果写入状态，再把新的观察送回模型继续推理。

理解 Agent 最稳固的方式，是用四要素模型。LLM 是大脑，负责语言理解、推理和决策；Tools 是四肢，让系统可以访问搜索、计算、数据库、文件和外部 API；Memory 是记忆，保存对话上下文、用户偏好、历史结果或长期知识；Planning 是规划层，把大目标拆成多步行动顺序，并在执行过程中调整。四者缺一不可。只有 LLM 没有工具时，它知道该做什么却做不了；只有工具没有规划时，它会像“随机点按钮”的系统；只有记忆没有筛选时，状态又会变成噪声源。

Agent 的自主性不是非黑即白，而是一条连续光谱。最左侧是“纯指令执行”: 模型只按你明确给出的步骤走，不主动扩展任务；往中间走，是“受约束的半自主”: 模型可以在限定工具集和预算内自己决定是否检索、是否分解子任务；更右侧则是“高自主执行”: 系统拥有更宽的权限、更多轮次和更少人工干预。工程上通常不追求越自主越好，因为自主性越高，收益和风险都会同步放大，尤其是成本失控、误操作和目标漂移。

典型场景能帮助你判断什么时候该上 Agent。个人助理强调多工具协同，比如查日程、读邮件、整理待办；数据分析型 Agent 需要拉取数据、运行代码、做中间校验；自动化运维更关心执行脚本、读取监控和处理告警；研究助手则常见于多轮搜索、文档阅读、证据整理和报告生成。共同点是: 任务不是一句话就能稳妥完成，而是需要跨步骤、跨工具、跨状态管理。

## 关键对比 / 易错点

- “会多轮聊天”不等于 Agent。关键区别是系统是否能围绕目标自主做下一步决策并调用外部能力。
- Agent 不只是 LLM 加工具。没有记忆和规划时，很多复杂任务仍然会退化成一次次零散调用。
- 自主性不是越高越先进。权限、预算和失败恢复机制没设计好时，高自主只会更快放大错误。
- 不是所有 AI 应用都该做成 Agent。一次性问答、固定提取、稳定分类任务，往往链路越简单越可靠。
- “模型回答得像在行动”不代表真的行动了。是否真正访问外部环境，要看系统执行层是否介入。

## 学习检查点

- 你能否用一句话说明纯 LLM 对话和 Agent 的最核心差别？
- 如果一个系统只有聊天接口和检索插件，但没有任务规划与状态管理，它更像完整 Agent 还是增强型问答系统？为什么？
- LLM、Tools、Memory、Planning 四个部分里，哪一个最容易在初学阶段被忽视？忽视后会出现什么症状？
- 面对“帮我调研某个领域并整理对比报告”这种任务，为什么 Agent 通常比单次 prompt 更合适？

## 实践建议

- 拿你熟悉的一个 LLM 应用，强制拆成“大脑、工具、记忆、规划”四个槽位，看看哪些组件其实还不存在，训练自己的系统化视角。
- 选一个你日常会做的重复任务，例如查资料、汇总信息或批量处理文件，判断它落在自主性光谱的哪个位置，再反推需要给 Agent 多大权限才合理。
