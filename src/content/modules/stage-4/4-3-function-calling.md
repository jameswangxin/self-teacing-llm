## 为什么学这个

如果说 ReAct 解决的是“Agent 怎么思考和决定下一步”，那么 Function Calling 解决的是“模型如何以机器可执行的方式表达这个决定”。只靠自然语言让模型输出“请帮我调用天气工具并传入北京”，工程上并不牢靠，因为字符串格式、参数命名和边界校验都不稳定。Function Calling 的意义，就是把工具调用从 prompt 习惯升级为结构化协议。

这一课直接影响你后面所有 Agent 实现质量。schema 写得清不清楚，会决定模型会不会选错工具、传错参数或根本不知道工具适用边界；`tool_choice`、并行调用和厂商协议差异，则决定你的系统能不能稳定控制模型行为。理解这一层后，你会把“工具接入”视为接口设计问题，而不是仅仅是 SDK 调用问题。

## 核心知识

OpenAI 风格的 Function Calling 通常包含四步闭环。第一步是开发者向模型声明工具列表，也就是函数定义，包括名称、用途说明和参数 schema；第二步是模型基于当前上下文决定是否要调用某个工具，以及传入什么参数；第三步由你的应用代码真正执行这个工具，例如发 HTTP 请求、查数据库、运行本地函数；第四步把工具返回结果作为新的消息回填给模型，让它继续推理或生成最终回答。关键点在于: 模型只负责“提议调用”，真正执行权仍在你的系统手里。

Function Schema 的质量决定了调用质量。`name` 要短、稳定、语义清晰，让模型知道这是干什么的；`description` 不只是给人看的注释，而是给模型做工具选择的重要提示，应该明确适用场景和限制；`parameters` 则要像 API 契约一样严谨，字段名具体、类型准确、必要字段清楚、枚举值尽量收敛。常见反模式包括: 工具名过泛、描述只写“获取信息”、参数字段叫 `input` 或 `data` 这种没有语义的名字，最后模型虽然调用了工具，但传参全靠猜。

`tool_choice` 体现的是控制权分配。自动选择模式适合大多数 Agent，让模型判断是否需要工具；强制指定某个工具，则适合你已经在上层逻辑中决定必须走某条路径，例如天气查询请求必须调用天气 API；显式禁止工具，则适合你只想要纯文本回答或做对比实验。理解这点很重要，因为很多工程问题不是模型不会调用，而是系统在不该放权时放得太多。

并行调用是另一个常见能力。某些模型支持在同一轮里提出多个独立工具调用，例如同时查询两个城市天气、并行拉取多个数据源。这能缩短整体延迟，但也提高了执行层复杂度: 你需要收集多个结果、处理部分失败、决定是否全部回填，甚至在顺序依赖存在时主动禁止并行。并行不是默认更高级，它只适合彼此独立、结果聚合清晰的任务。

跨模型厂商时，协议细节也会不同。OpenAI 常见的是 `tools` 与结构化参数对象，Claude 更强调 tool use content block 和消息块级别的往返，DeepSeek 等 OpenAI-compatible 服务在接口形态上可能接近 OpenAI，但在字段支持、并行行为或错误返回上仍有差异。工程上应该把“工具抽象”和“供应商协议适配”分开，不要把业务工具定义和某家 SDK 的消息格式硬绑在一起。

## 关键对比 / 易错点

- Function Calling 不是模型直接执行代码，而是模型生成结构化调用意图，执行仍由你的应用负责。
- schema 不是越宽松越友好。描述模糊、参数随意，模型反而更容易误选和误传。
- `tool_choice=auto` 并不等于“最智能”，它只是把是否调用工具的决定权交给模型。
- 并行调用只适合互相独立的任务；有前后依赖时强行并行，常常会得到无效结果。
- “接口兼容 OpenAI”不代表所有工具调用语义都完全一致，跨厂商时必须单独验证。

## 学习检查点

- 为什么说 Function Calling 把工具使用从“自由文本提示”变成了“接口契约”？
- `name`、`description`、`parameters` 三部分里，哪一部分最直接影响模型是否选对工具？为什么？
- 如果你已经在业务层判断必须访问数据库，这时 `tool_choice` 更适合用自动还是强制模式？
- 并行工具调用带来的收益和新增复杂度分别是什么？

## 实践建议

- 为一个简单工具重写 schema，不要使用含糊字段名，强迫自己把每个参数都写成别人不看实现也能理解的契约。
- 用同一个工具集分别接 OpenAI-compatible 接口和另一家模型服务，记录消息格式、并行支持和错误处理差异，建立协议适配意识。
