## 为什么学这个

很多 Agent Demo 看起来“会调用工具”，但一到真实环境就暴露出大量问题: 工具名字含糊、描述太短、返回结果不可读、报错一团糟，导致模型不是不用工具，就是用错工具，或者一失败就彻底卡住。问题往往不在模型，而在工具本身没有被设计成适合给模型使用的接口。

这一课要建立的是工具工程观，而不是“把一个 Python 函数包上去就完事”。Agent 工具不是普通内部函数。它既服务于执行层，也服务于模型决策层，所以接口说明、失败语义和返回格式都要面向模型优化。先把通用工具设计原则想清楚，后面不管你接 LangChain、自己写执行器，还是换成别的 Agent 框架，判断标准都不会变。

## 核心知识

设计工具时，第一优先级不是“用什么框架封装”，而是先定义清楚工具契约。一个可供 Agent 使用的工具，至少要回答四个问题: 它解决什么问题、什么时候该用它、输入参数长什么样、输出结果会返回什么。你可以把工具理解成给模型暴露的一段受控能力接口。模型不会读你的实现代码，它只会根据你提供的契约来决定是否调用。因此，一个工具如果边界模糊，即使底层逻辑写得再完整，也很难被正确使用。

工具描述的重要性经常被低估。模型选工具时，并不会读你的代码实现，只能看到名称、描述和参数说明。一个好的描述应该告诉模型“什么时候该用我、不该用我时怎么办、输入应该长什么样、输出大致是什么”。例如搜索工具要说明适合查最新公开信息，不适合回答纯数学问题；数据库工具要说明只查哪些表、是否需要精确 ID；文件工具要说明路径范围和权限限制。描述写得越模糊，模型越容易把工具当成万能入口，最后把本该直接回答的问题也错误路由到工具层。

常见工具类型包括搜索、计算、数据库、API、文件五大类。搜索工具适合最新信息和开放网页；计算工具适合精确数值和单位转换；数据库工具面对结构化内部数据；API 工具负责连接业务系统；文件工具则处理读写、解析和批处理。不同类型的关注点并不一样。搜索工具更重查询构造与结果摘要，数据库工具更重安全约束和返回规模，文件工具则更重路径校验和副作用控制。把它们统称成“工具”很方便，但工程上不能用同一套粗糙标准处理。

失败处理和结果格式化，决定 Agent 是否有恢复能力。工具可能超时、返回空数据、参数不合法、权限不足，或者上游 API 结构变化。好的工具不会把一长串堆栈直接扔给模型，而会返回模型可理解的失败信息，例如“未找到匹配记录，请提供更具体的订单号”。同样，结果也不该是未经处理的原始 JSON 洪流。你要根据后续推理需要，把结果整理成简洁字段、限制长度、保留关键证据，这样模型才能基于 Observation 做出更稳定的下一步判断。

到了真正实现时，才轮到框架选择。像 LangChain 这样的框架会提供 `@tool` 或 `BaseTool` 之类的封装方式，帮助你把上述契约、执行逻辑和错误处理接入 Agent 运行时。这里要反过来理解: 框架 API 只是承载工具设计的一种实现路径，不应该代替你思考工具边界本身。如果契约没设计好，换任何框架都只是在更快地暴露问题。

## 关键对比 / 易错点

- 工具是否好用，更多取决于描述、输入契约和失败语义，而不是实现代码写得多炫。
- 先想清楚“工具该长什么样”，再决定用哪个框架封装；不要把框架 API 当成工具设计本身。
- `@tool` 或 `BaseTool` 这类封装只是实现手段，不是课程这一讲的核心抽象。
- 工具返回“越完整越好”是常见误区。给模型一大坨原始数据，通常会增加噪声和误读。
- 报错直接透传底层异常，对开发者可能有用，对模型通常没有用；要返回可行动的失败信息。
- 不同工具类型的风险不同，文件和数据库工具尤其要注意权限边界与副作用。

## 学习检查点

- 为什么说工具描述是模型选择工具时最重要的上下文之一？
- 为什么说工具契约设计应该先于框架选型？
- 如果后面要把同一个工具接入不同 Agent 框架，哪些设计部分应该保持稳定？
- 如果搜索工具返回几十条原始网页片段，为什么这不一定比返回 3 条结构化摘要更好？
- 当工具失败时，什么样的错误信息更有助于 Agent 继续决策？

## 实践建议

- 选一个最小工具，例如计算器或搜索器，同时写一版“糟糕描述”和“一版高质量描述”，对比模型调用行为差异，训练工具说明写作能力。
- 为你后续要做的 Agent 工具预先定义统一返回格式，例如 `status`、`message`、`data` 三段式，这样后面接多种工具时更容易做恢复和调试。
