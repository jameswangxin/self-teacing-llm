## 为什么学这个

很多 AI 应用在原型阶段看起来没什么成本问题，因为调用量小、用户少、数据规模有限。但一旦进入真实业务，延迟、吞吐和费用会迅速变成核心约束。你会发现回答再准，如果每次都慢 15 秒、每月账单不可控、并发一高就排队，系统仍然没法落地。

性能与成本优化的难点，在于它不是单点技巧，而是一整套工程组合拳。学这一课，是为了建立更现实的判断: 什么该缓存，什么该异步，什么时候该路由到小模型，什么时候本地部署反而更合适，如何在质量和资源约束之间做持续折中。

## 核心知识

缓存通常是第一层最划算的优化，但不同缓存解决的是不同问题。Embedding cache 缓存的是文本到向量的结果，适合重复索引或重复查询嵌入，减少重复计算；semantic cache 缓存的是“相似问题 -> 可复用答案或中间结果”，它依赖相似度阈值，风险在于把“看起来像”的问题错当成“可以直接复用”的问题；LLM 响应缓存更直接，命中条件往往是请求完全一致或高度规范化，最适合模板化任务、固定问法和批量重复操作。缓存不是越多越好，关键是弄清楚命中条件和失效策略。

Token 预算控制是另一条主线。很多系统成本高，不是模型太贵，而是无节制地塞入长历史、冗余上下文和重复检索结果。你需要从 prompt、上下文拼装、检索召回数、摘要压缩和回答长度几层一起下手，明确哪些 token 真正在提高质量，哪些只是因为“怕漏”才被堆进去。预算控制本质上是信息密度优化，而不是简单把字数砍掉。

异步与批处理解决的是吞吐问题。非关键路径上的 embedding、评估、索引更新、日志写入和部分工具调用，可以通过异步化减少用户等待；批处理则适合离线生成 embedding、大批量文档处理、批量评估或统一调用模型接口。这里要注意，批处理虽然便宜，但会引入等待窗口和失败重试复杂度，不一定适合强实时任务。优化的目标不是“全都异步”，而是把用户必须等待的部分和后台可延后的部分分开。

模型路由是质量与成本之间最常见的分层手段。简单分类、提取、摘要和规则化改写任务，常常可以用更便宜的小模型；复杂推理、长上下文整合和高风险输出再交给大模型。真正困难的地方不是写一个 if/else，而是建立稳定的路由准则和回退策略，避免“本来是省钱，结果因为误路由导致质量不稳、还要二次调用更贵”。

本地部署 awareness 也属于生产优化的一部分。Ollama 适合本地开发、轻量实验和单机模型管理；vLLM 更偏向高吞吐推理服务，适合需要连续批处理、并发优化和更可控部署环境的团队。它们的意义不只是省 API 费，还涉及数据不出域、延迟控制和基础设施掌控权。但本地部署会把问题从“调用外部接口”变成“自己承担模型服务运维”，这笔账要算清楚。

## 关键对比 / 易错点

- Embedding cache、semantic cache 和响应缓存缓存的对象不同，混用会导致错误命中和过时结果。
- 成本问题常常是上下文膨胀造成的，不一定是模型单价本身最致命。
- 异步和批处理能提吞吐，但不自动等于更好的用户体验；实时链路和离线链路要分开设计。
- 小模型路由不是默认正确，路由判断错误会把成本优势吃回去，甚至损害稳定性。
- 本地部署不是“零成本替代”，它把隐性成本转移到了运维、硬件和模型服务治理上。

## 学习检查点

- Embedding cache、semantic cache 和 LLM 响应缓存分别适合缓存什么？
- 为什么说 token 预算控制的核心是信息密度，而不是机械缩短 prompt？
- 哪些工作更适合异步或批处理，哪些必须保留在实时链路中？
- 在什么情况下，小模型路由会节省成本；在什么情况下反而会制造返工？

## 实践建议

- 先对你当前系统的单次请求做一次成本拆账，分别记录检索、上下文长度、模型调用次数、工具耗时和缓存命中率，再决定先优化哪一层。
- 实现一个最小路由实验，把“简单提取/分类”与“复杂推理/综合回答”分给不同模型，并记录质量差异与费用变化，不要直接在全链路上线。
