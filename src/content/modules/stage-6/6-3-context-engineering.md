## 为什么学这个

很多人一谈 AI 应用优化，第一反应还是“prompt 还不够好”。这在 Demo 阶段没错，但一旦系统开始接入检索、记忆、工具结果、用户历史和业务规则，决定质量的往往不再是某一句提示词，而是模型最终看到了哪些信息、这些信息以什么顺序出现、哪些被裁掉了。问题已经从 Prompt Engineering，转向 Context Engineering。

Context Engineering 的价值，是把“喂给模型的信息”视为一个可设计系统，而不是临时字符串拼接。学这一课，是为了建立一个更贴近生产的意识: 大模型的能力上限，很大程度取决于你如何管理上下文预算、选择信息来源、按任务动态组装上下文。

## 核心知识

Prompt Engineering 更关注单条提示的措辞、结构和约束，适合模型输入相对固定的场景；Context Engineering 关注的是完整输入包如何被构成。它会追问: 系统指令写了什么，用户原始问题是什么，历史对话要保留多少，检索片段该进哪些，工具返回是全文塞入还是只取关键信息，记忆是近期状态还是长期偏好。换句话说，prompt 是上下文的一部分，而不是全部。

在生产系统里，可以把上下文来源先分成四类。第一类是系统层信息，例如角色设定、业务规则、输出格式和安全约束；第二类是用户层信息，包括当前问题、用户属性和目标；第三类是状态层信息，例如会话历史、Scratchpad、工作流节点状态和长期记忆；第四类是外部证据层信息，如检索结果、数据库记录、工具调用返回和实时环境数据。很多回答质量问题，根源不是模型差，而是这四类来源混在一起、优先级不清，导致模型读到了不该读的，错过了最该读的。

上下文预算管理是工程核心。上下文窗口再大也不是无限的，而且越长越贵、越慢、越容易让重要信息被淹没。预算管理并不等于简单截断，而是要做优先级分层: 哪些信息必须完整保留，哪些可以摘要，哪些只在特定任务注入，哪些在当前轮次应该完全丢弃。一个成熟系统常常会先给系统规则、用户目标、关键状态和检索证据分别设预算上限，再用压缩、摘要、去重和排序把有限 token 留给最有效的信息。

动态上下文组装意味着不同问题走不同的上下文路径。查询产品文档时，重点可能是最新版本说明和命中的章节；做数据分析时，重点是工具结果和中间状态；执行 Agent 工作流时，还要考虑上一步失败原因、当前节点状态和工具权限。Context Engineering 的成熟度，体现在系统是否能根据任务类型动态选择信息，而不是永远把“所有可能相关的内容”一次性塞进去。

## 关键对比 / 易错点

- Prompt Engineering 没有过时，但它只解决“怎么说”，Context Engineering 还要解决“给什么”和“给多少”。
- 上下文越长不代表越好，信息过载会让关键约束被稀释，甚至增加幻觉和跑偏概率。
- 会话历史不是越多越安全，很多旧轮次信息会制造冲突或把当前任务带偏。
- 动态上下文组装的目标不是复杂化，而是让不同任务只看到最必要的信息。
- 如果没有明确预算和优先级，系统最后往往退化成“随机拼接一大段文本”。

## 学习检查点

- Prompt Engineering 和 Context Engineering 的关注点分别是什么？
- 生产系统常见的四类上下文来源是什么？它们为什么不能无差别混用？
- 为什么说上下文预算管理不是单纯删字，而是信息优先级设计？
- 如果一个 Agent 总是在长上下文里忽略关键工具结果，可能是哪类上下文工程问题？

## 实践建议

- 选一个你现有的 RAG 或 Agent 流程，画出模型每一轮真实看到的上下文组成，标注每块内容的来源、长度和作用。
- 给系统层规则、用户问题、检索结果和历史状态分别设置 token 预算，做一次手工压缩实验，观察回答是否因为上下文重排而明显改善。
