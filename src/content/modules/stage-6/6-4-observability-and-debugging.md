## 为什么学这个

生产级 AI 系统最大的错觉之一，是“本地跑通几次就说明系统没问题”。真正上线后，你会遇到用户说回答变慢了、某类问题突然答偏、某个工具今天频繁失败、某条工作流陷入循环，但如果系统没有 tracing、评估和监控，你根本不知道问题出在检索、模型、工具还是上下文拼装。

可观测性和调试不是锦上添花，而是你是否能长期维护 AI 系统的分水岭。学这一课，是为了把“出故障后手工猜原因”升级成“有轨迹、有指标、有回放、有方法论”的排障习惯，让系统从 Demo 变成可运维的产品。

## 核心知识

Tracing、evaluation、monitoring 解决的是三类不同问题。Tracing 关注单次请求内部发生了什么，帮助你看到 prompt、检索、工具调用、状态变化和每一步耗时；evaluation 关注系统质量是否达标，回答的是“这套链路现在答得好不好”；monitoring 关注线上整体健康度，例如延迟、错误率、成本、命中率和异常峰值。三者缺一不可。只有 tracing 没有评估，你知道发生了什么却不知道结果好不好；只有监控没有 trace，你知道坏了却不知道坏在哪。

LangSmith 和 Langfuse 是常见的可观测性平台，但关注点略有差异。LangSmith 更强调链路追踪、调试体验和与 LangChain/LangGraph 生态的紧密集成，适合快速查看调用树、输入输出、评估结果和实验对比。Langfuse 则更强调更通用的 observability 视角，常用于 trace、评分、成本和反馈闭环管理。你不必把它们神化为唯一答案，重点是理解平台该帮你保留哪些证据: 请求级 trace、步骤级输入输出、用户反馈、成本统计、失败样本和评估结果。

Agent 调试要有顺序，而不是看到输出错就直接改 prompt。更可靠的方法通常是: 先重现问题并固化样本，再看 trace 判断故障发生在哪一层，然后分别排查输入是否失真、检索是否跑偏、工具参数是否错误、模型决策是否不稳定、状态是否被污染。只有定位到了故障层级，改动才有意义。否则你往往是在系统的错误症状上贴创可贴。

成熟团队会把调试和评估连起来。线上监控发现异常后，要能快速抽样到失败 trace；从失败 trace 中整理出可复现案例，再进入离线评估集回归；修复后重新跑评估，确认不是只治好了某一个样本。这种“监控发现问题 -> trace 定位原因 -> 评估验证修复”的闭环，才是 AI 系统调试真正稳定的方式。

## 关键对比 / 易错点

- 日志不是 trace 的完全替代，散落的控制台输出很难还原完整调用树和状态流。
- tracing、evaluation、monitoring 不要混成一个概念，它们解决的问题层次不同。
- 只盯最终答案对错，会错过很多“路径已经异常但恰好答对”的隐患。
- 调试 Agent 时先定位故障层，再决定改 prompt、改工具、改状态还是改工作流。
- 没有失败样本沉淀时，每次线上事故都会变成一次性的经验，难以形成回归机制。

## 学习检查点

- tracing、evaluation、monitoring 分别最适合回答什么问题？
- 为什么说“看到了错误答案”还不足以指导修复？
- LangSmith 或 Langfuse 这类平台在工程上真正提供的核心价值是什么？
- 如果一个 Agent 偶发地走错工具，你会按什么顺序做定位？

## 实践建议

- 给你当前的一个 RAG 或 Agent 原型接入最基本的 trace 记录，至少保留输入、检索结果、工具参数、模型输出和总耗时。
- 从最近 5 个失败案例里挑 2 个做完整复盘，写出“故障层级、证据、根因、修复动作、回归验证方式”五项记录，训练自己的调试方法论。
