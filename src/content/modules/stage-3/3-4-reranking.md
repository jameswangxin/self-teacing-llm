## 为什么学这个

初始检索的任务更像“粗筛”: 在大语料里快速找出一批可能相关的候选。它追求的是速度和覆盖，不是最终精度。问题在于，真正放进 prompt 的上下文预算非常有限，如果你直接按初始检索结果塞进去，常常会把“有点相关”的片段和“真正关键”的证据混在一起，导致答案质量上不去。

重排序的意义，就是在召回之后再做一次更精细的相关性判断。你可以把它理解成二段式检索: 第一段负责别漏掉，第二段负责把最值得读的内容顶上去。理解这一步后，你会更清楚为什么很多生产级 RAG 系统不是“检索一次就结束”，而是把性能预算留给最有价值的几十个候选。

## 核心知识

之所以需要 reranking，是因为向量检索和 BM25 都有先天局限。它们在海量语料上做近似匹配很高效，但对“这个具体问题和这段文档合在一起到底有多相关”判断得不够细。尤其当候选文档彼此都差不多相关时，初始分数往往不足以决定谁应该进入最终上下文。二次排序就是拿更贵但更准的模型，在更小候选集上做精排。

Bi-encoder 和 cross-encoder 是这里最常见的两种架构。bi-encoder 先分别把查询和文档编码，再用向量相似度比较，速度快、适合大规模检索；cross-encoder 则把查询和文档一起送进模型，让模型直接判断这对文本的相关性，因此通常更准，但也更慢。工程上常见模式是“bi-encoder 负责大规模召回，cross-encoder 负责小范围精排”，这也是为什么 reranker 通常出现在初检之后。

具体工具上，Cohere Reranker 通常以易用 API 和稳定效果见长，适合希望快速接入托管服务的团队；BGE-Reranker 常见于开源、自部署和中文场景；FlashRank 则更偏轻量级、本地快速部署。选型时不要只看榜单分数，还要看延迟、成本、是否可私有化、语言支持以及和现有链路的集成复杂度。

重排序还有一个重要价值是缓解 lost in the middle。与其把 20 条候选原样塞给模型，不如先用 reranker 把真正关键的 3 到 5 条顶出来，再决定是否做 `ContextualCompressionRetriever`。压缩检索器会在保留相关片段的同时去掉无关内容，进一步降低上下文噪声。这里要记住，reranking 解决的是“候选排序”，compression 解决的是“片段体积和噪声”，两者经常搭配，但不是同一个动作。

## 关键对比 / 易错点

- 初始检索负责高召回，不负责最终精排；把第一阶段结果直接当最终上下文，通常浪费预算。
- bi-encoder 更快，cross-encoder 更准。不要用一个标准同时要求它们兼顾海量检索和极致精排。
- reranker 不会凭空找回漏召回的文档，它只能在已有候选里重新排序。
- `ContextualCompressionRetriever` 不是 reranker 的替代品，它更像在精排后继续删噪和裁剪。
- 候选集太大时，reranking 本身也会变贵，所以通常先控制初检 Top-K，再做精排。

## 学习检查点

- 为什么说 reranking 解决的是“有限上下文预算分配”问题，而不只是“让排名更好看”？
- bi-encoder 和 cross-encoder 在工作方式、速度和精度上有什么根本差异？
- 如果关键文档根本没有进入初始候选集，加入 reranker 能不能解决？为什么？
- lost in the middle 和 `ContextualCompressionRetriever` 之间是什么关系？

## 实践建议

- 对同一批查询分别保存“初始 Top-K”和“rerank 后 Top-K”，手动比较哪些文档被顶上来，判断重排序是否真的有用。
- 如果你后续实现 reranker，先在小候选集上试验延迟和效果，再决定是否要进一步叠加 compression，而不是一次把链路做得过重。
