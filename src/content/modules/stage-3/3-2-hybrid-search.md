## 为什么学这个

只靠向量检索做 RAG，早期 Demo 常常看起来很聪明，因为它对语义相近表达有不错容错。但一旦进入真实业务，你很快会碰到它的短板: 型号名、错误码、函数名、缩写、日期、合同编号这些“必须精确命中”的内容，经常会被语义近似但字面不同的文档抢掉排名。

混合检索的价值，就是把“语义匹配”和“关键词精确匹配”放进同一条召回链里。你要学会判断什么时候向量检索已经够用，什么时候必须引入 BM25 或全文检索，再理解怎么把两路结果融合。这样后面做企业知识库、代码搜索和规范文档问答时，召回质量才不会卡在底层。

## 核心知识

向量搜索的优势是能理解改写、近义词和语义相关性，但它并不天然擅长精确 token 匹配。比如用户问某个错误码、产品 SKU 或函数签名时，向量空间里“意思差不多”的文本可能比“字面完全命中”的文本排得更高，导致看起来像相关、其实不能直接回答的问题。理解这个短板很重要，因为它解释了为什么很多系统在通用问答表现不错，但在技术文档和规章制度场景里经常漏关键细节。

BM25 可以看成是经典关键词检索的代表。它关心某个词在当前文档里出现得多不多，在整个语料库里稀不稀有，以及文档本身是否过长。直觉上，稀有且被问题明确提到的词，权重会更高，所以 BM25 对专有名词、数字和短语匹配特别有价值。它的弱点则是语义泛化差，用户换一种说法时，可能完全匹配不到。

Hybrid Search 的核心不是二选一，而是让两种召回方式互补。常见做法是并行跑向量检索和 BM25，再把两个结果列表融合。RRF，也就是 Reciprocal Rank Fusion，是一种简单但稳定的融合方法: 不直接比较不同系统的原始分数，而是按各自排名给文档打分，排名越靠前贡献越高。它的好处是实现简单，对分数尺度不敏感，特别适合把来自不同检索器的结果合并成一个候选集。

在 LangChain 里，`EnsembleRetriever` 就是常见实现路径。你可以把一个向量检索器和一个 BM25 检索器放进去，给它们设置权重，再统一返回融合后的结果。工程上真正需要调的不是“有没有混合检索”这么简单，而是每一路召回各取多少候选、RRF 或权重怎么设、哪些查询场景只走某一路，以及怎样在日志里观察两路召回到底谁在贡献命中。

## 关键对比 / 易错点

- 向量检索强在语义泛化，不强在精确关键词命中；BM25 正好相反，二者不是替代关系。
- Hybrid Search 不是把两路结果简单拼接。没有融合策略时，重复项和低质量候选会迅速放大噪声。
- RRF 的优势在于不用强行对齐不同检索器的分数尺度，很多场景下比“硬拼分数”更稳。
- `EnsembleRetriever` 只是组合器，不会自动替你决定权重、候选数和何时退化成单一路召回。
- 如果语料很干净、问题也偏自然语言，混合检索不一定总有收益；它主要解决向量检索的盲区。

## 学习检查点

- 为什么向量检索在错误码、产品型号、函数名这类查询上容易失手？
- 你能否用自己的话解释 BM25 为什么会偏爱“稀有且被明确提到”的词？
- RRF 融合时为什么更关注排名而不是原始分数？这解决了什么问题？
- 如果你在 LangChain 里要做混合检索，`EnsembleRetriever` 在整条链路里扮演什么角色？

## 实践建议

- 选一批包含专有名词和一批包含自然语言改写的问题，分别比较纯向量检索、纯 BM25、混合检索的 Top-K 命中情况。
- 给你的检索日志加上“命中来自向量、BM25 还是两者同时命中”的标记，后续调权重时才知道改动是否真的有效。
