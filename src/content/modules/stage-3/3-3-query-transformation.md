## 为什么学这个

用户提问方式很少像知识库那样规整。真实输入经常是口语、省略、带背景假设，甚至一次把两个问题揉在一起。如果你直接把原始输入丢给检索器，系统并不是在“理解用户”，而是在被迫拿一个噪声很大的查询去做近似匹配，结果自然不稳定。

查询优化的价值，在于把“用户表达”转成“检索表达”。它不是为了让问题看起来更高级，而是为了提升召回率、覆盖不同表述并拆开复杂意图。学会这套方法后，你就能系统处理“问法不标准”这类常见工程问题，而不是把所有锅都甩给 embedding 或向量库。

## 核心知识

最基础的方法是 query rewriting，也就是先让模型把用户问题改写成更清晰、更接近知识库措辞的查询。例如把“退款规则里关于企业客户的那部分怎么说”改成“企业客户退款政策 条件 流程”，或者把带代词的问题还原成完整实体。它的目标不是生成更多内容，而是减少歧义、补足关键限定词，让检索器更容易命中正确文档。

Multi-Query Retrieval 则是另一条思路: 不只改写成一个更好的问题，而是生成多个不同角度的检索查询，再把各自结果合并。它特别适合用户问题有多种表达方式、知识库写法不统一的场景。代价是召回候选数和成本会上升，因此通常要配合去重、重排序和结果裁剪，否则很容易把“扩召回”做成“扩噪声”。

HyDE，也就是 Hypothetical Document Embeddings，是先让模型根据问题写出一段“假想答案”，再拿这段答案去做向量检索。背后的直觉是，知识库文档往往不像用户提问，而更像一段说明性内容，所以用假想答案的语义形态去搜，有时比原问题更贴近真实文档。Step-Back Prompting 则是先把具体问题提升到更抽象的一层，例如从“这个 API 为何报 401”退一步改成“身份认证失败的常见原因”，用于补充更一般的背景材料。

更复杂的问题则需要 decomposition。用户一句话里可能其实包含多个子问题，或者答案必须经过多跳检索和组合才能完成。例如“比较 A 方案和 B 方案在成本、部署要求和更新机制上的差异”，如果只做一次整体检索，常常拿不到足够完整的证据。把问题拆成多个子查询，分别召回后再汇总，通常更稳。要注意的是，查询优化本身也会引入漂移: 改写得太自由，可能把原问题改坏；生成太多查询，也可能把无关候选带进来，所以必须靠日志和评估反向约束。

## 关键对比 / 易错点

- query rewriting 的目标是降歧义，不是把问题“润色得更长”。更长不等于更好检索。
- Multi-Query 是扩覆盖面，Decomposition 是拆任务结构，两者解决的不是同一个问题。
- HyDE 更像“用假想文档形态去搜”，适合向量检索；BM25 场景下收益通常没那么直接。
- Step-Back Prompting 能补背景，但如果问题本身已经很具体，退得太远会稀释检索焦点。
- 查询改写链路必须可观察。否则你只看到最终答案变差，却不知道是原问题、改写问题还是融合逻辑出了问题。

## 学习检查点

- 什么情况下 query rewriting 会比直接检索原问题更有效？
- Multi-Query Retrieval 和 Query Decomposition 在工程目标上有什么本质区别？
- HyDE 为什么要先生成一段“假想答案”，这对向量检索有什么帮助？
- 如果改写后的查询开始频繁命中不相关文档，你会优先从哪些地方排查？

## 实践建议

- 为同一批问题同时记录“原始查询、改写查询、多查询列表、最终召回结果”，观察哪种策略真正带来了命中提升。
- 先挑 5 个明显模糊或复合型问题，分别试验 rewriting、multi-query、decomposition，建立各方法的适用直觉再决定是否上线。
