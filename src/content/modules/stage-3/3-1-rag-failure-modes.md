## 为什么学这个

很多团队第一次把 RAG 跑通后，会陷入一种危险错觉: 只要“检索到了文档”，系统就应该稳定回答正确。真实情况正好相反。RAG 的问题往往不是单点故障，而是召回、排序、上下文注入和答案生成一起作用后的复合结果。如果你不会区分失败类型，后续优化就容易变成盲调参数。

这一课的目标是先建立诊断框架。你要能看着一次失败案例回答三个问题: 到底是没找到证据、找到了但模型没忠实使用、证据太多被淹没了，还是用户问题本身就不够明确。只有把失败模式映射清楚，后面学混合检索、查询改写、重排序和评估时，才知道每个方法在解决哪一类问题。

## 核心知识

RAG 的第一类典型故障是 recall failure，也就是系统根本没有把关键证据召回出来。表现通常是检索结果一眼看去就不相关，或者只召回了边缘信息，真正能回答问题的片段没有进入候选集合。原因可能出在分块不合理、嵌入模型不匹配、向量检索对精确术语不敏感，或者用户问题和知识库表达方式差异太大。遇到这种问题，优先检查检索结果本身，而不是先改生成 prompt。

第二类是 faithfulness failure。这里检索到的材料其实已经足够，但模型回答时曲解了证据、补了文档里没有的细节，或者把多个片段混成了错误结论。这种失败说明问题不在“有没有材料”，而在“有没有忠实使用材料”。常见改进包括更明确的引用式 prompt、减少无关上下文、加入重排序和压缩，以及在评估中单独测 faithfulness，而不是只看最终答案像不像对的。

第三类是 lost in the middle。很多人以为 Top-K 提高就一定更安全，实际上上下文一长，模型会更容易忽略中间位置的关键证据，只记住开头和结尾的信息。你看到的症状常常是“答案里提到了检索文档，但漏掉中间那条真正关键的限制条件”。这类问题通常需要控制上下文长度、做 reranking、压缩和动态 Top-K，而不是继续无上限地塞材料。

第四类是 query ambiguity。用户的问题可能本身就带有省略、代词、多意图或背景缺失，例如“这个政策什么时候生效”里的“这个政策”在系统视角并不明确。此时检索器不是坏了，而是在拿一个模糊问题做确定性匹配。对应的优化方向是 query rewriting、history-aware retrieval、必要时反问澄清，以及为不同失败模式建立一张“优化地图”: 召回问题优先看检索策略，忠实性问题优先看上下文和生成约束，上下文过载问题优先看排序与压缩，问题模糊则优先看查询改写与对话管理。

## 关键对比 / 易错点

- “答案错了”不等于“检索错了”。先分离检索结果和最终回答，才能知道问题落在哪一层。
- recall failure 和 faithfulness failure 常被混为一谈。前者是没找到证据，后者是有证据但没用好。
- Top-K 不是越大越稳。候选文档过多会带来 lost in the middle 和噪声污染。
- 用户提问口语化或带代词时，直接拿原问题检索常常不够，应该先考虑 query rewriting 或澄清。
- 优化顺序很重要。没有诊断就直接上复杂链路，通常只是把问题藏得更深。

## 学习检查点

- 你能否分别举例说明 recall failure、faithfulness failure、lost in the middle 和 query ambiguity？
- 如果检索结果已经包含正确证据，但答案仍然编造细节，你会把问题归到哪一类？优先怎么改？
- 为什么把 Top-K 从 5 调到 20 有时会让结果更差，而不是更好？
- 当用户问题里出现“它”“这个版本”“刚才那条规则”这类代词时，你会优先检查哪种优化手段？

## 实践建议

- 找一个现有 RAG Demo，连续记录 5 次失败回答，并把每次失败强制归类到某一种模式，训练自己的诊断习惯。
- 后续调参时，先保存“用户问题、检索 Top-K、最终答案”三联日志，再决定要改召回、排序还是 prompt，避免盲调。
