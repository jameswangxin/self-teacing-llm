## 为什么学这个

RAG 系统最容易陷入的工程陷阱之一，就是靠“看起来还行”做优化决策。今天换了一个 embedding，明天加了 reranker，后天改了 query rewriting，大家各自拿几个案例试一下，觉得似乎更好了就上线。这样做的问题不是不努力，而是没有稳定、可复现的质量基线，回归退化时你甚至不知道是哪一步引起的。

评估体系的作用，就是把主观体感变成客观证据。你要学会把问题拆成检索质量、答案忠实性和回答相关性几个维度，再用固定数据集和自动化流程重复验证。只有这样，RAG 优化才不再是“玄学调参”，而是可比较、可回滚、可持续演进的工程系统。

## 核心知识

RAG 评估至少要分三类核心指标。Context Relevance 关注检索出来的上下文与问题是否相关，回答的是“你给模型的材料对不对”； Faithfulness 关注答案是否忠于给定上下文，回答的是“模型有没有乱说”； Answer Relevance 则关注最终答案是否真正回应了用户问题，回答的是“就算忠于上下文，它有没有答到点上”。把这三类指标拆开，你才能知道问题是出在召回、生成还是用户意图匹配。

RAGAS 和 DeepEval 是两类常见的自动评估工具。RAGAS 更强调围绕 RAG 场景的一组指标组合，方便快速搭建评估基线；DeepEval 更像通用 LLM 评估框架，便于把多类评测组织进自动化测试流程。你不必把它们神化成“最终裁判”，因为很多指标本身仍依赖 LLM-as-a-judge，但它们至少能帮助你把评估流程标准化，而不是每次临时人工抽样。

评估数据集建设比选哪个框架更重要。一个好数据集要覆盖真实问题分布，而不是只收录最容易命中的漂亮样例。它应包含常见问法、模糊提问、多跳问题、带精确术语的问题、带时间或版本限制的问题，以及已知容易失败的边界案例。最好同时保存标准问题、参考答案、期望证据片段和任务标签，这样后续分析退化时，才能定位到底是哪一类问题开始变差。

自动化回归测试是把评估体系真正落地的最后一步。每次改 embedding、chunking、reranker、query transformation 或 prompt，都应重新跑固定评估集，比较关键指标和代表性案例。这里的重点不是追求单次绝对高分，而是建立“改动后有没有明显退化”的预警机制。生产级团队常见做法是把评估跑进 CI 或离线批处理里，让检索和回答质量像单元测试一样可以持续回归。

## 关键对比 / 易错点

- 最终答案看起来不错，不代表检索和忠实性都没问题；不同指标要拆开看。
- LLM-as-a-judge 很有用，但不是绝对真理；关键评估集仍需要人工校准和抽检。
- 数据集如果只包含简单样例，会让系统看起来稳定，实际上对真实难题毫无约束。
- 回归评估关注的是改动前后相对变化，不只是某次跑分的绝对高低。
- 没有记录期望证据片段时，很多失败案例只能看到“答错了”，却难以定位是召回错还是生成错。

## 学习检查点

- Context Relevance、Faithfulness 和 Answer Relevance 分别在判断什么问题？
- 为什么说一个高质量的评估数据集，比单纯换一个更时髦的评测框架更重要？
- RAGAS 和 DeepEval 在你的理解里更像解决什么工程问题？
- 如果某次改动后答案相关性上升了，但 faithfulness 下降了，你会如何解读这个结果？

## 实践建议

- 先从 20 到 30 个真实用户问题开始，手工整理一个最小评估集，至少标注问题、参考答案、期望证据和问题类型。
- 把每次检索链路改动都绑定一次固定评估运行，哪怕一开始只是本地脚本，也要建立“改动必回归”的纪律。
