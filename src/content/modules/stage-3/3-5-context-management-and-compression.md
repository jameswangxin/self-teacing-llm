## 为什么学这个

RAG 做到后面，很多问题不再是“找不到”，而是“找到太多”。文档越来越长，候选越来越多，上下文窗口再大也不是免费的。你如果没有上下文管理意识，系统就会在成本、延迟和答案稳定性之间同时失控: 模型读得慢、噪声变多、关键片段反而被淹没。

这一课关注的是怎样让进入 prompt 的每一段内容都更值钱。核心不是盲目压缩文本，而是控制信息密度和相关性。你要学会什么时候该删、什么时候该保留父文档上下文、什么时候先做 metadata 过滤，以及如何根据问题难度动态调整 Top-K，而不是一套参数打天下。

## 核心知识

Contextual Compression 的目标是把“候选文档”压缩成“回答当前问题真正需要的证据”。它可以通过抽取关键句、截取相关片段或剔除无关段落来实现。重点不是追求最短，而是追求更高的信噪比。压得过头会丢失约束条件，压得太保守又无法缓解上下文膨胀，所以压缩策略必须围绕具体问题来判断保留什么。

Parent-Child 检索是处理“检索粒度”和“阅读粒度”矛盾的常见方法。子块适合做精确召回，因为它短、小、语义集中；父块或原始段落适合给模型阅读，因为它保留了更完整的上下文。典型做法是先用 child chunk 命中，再回溯到 parent document 作为最终提供给模型的内容。这样可以兼顾召回精度和语义完整性，避免短块命中了但信息支离破碎。

Metadata 过滤则是更便宜的第一道缩圈手段。时间范围、文档类型、业务线、语言、权限级别、产品版本，这些元数据如果在索引阶段标得好，查询时就能先过滤一大批不可能相关的文档。它的价值不只是提速，更是减少错误证据混入候选集。很多团队忽略这一层，结果用昂贵的向量检索去解决本来可以靠结构化筛选提前排除的问题。

动态 Top-K 解决的是“不同问题需要不同证据量”。简单事实问答可能 3 条证据就够，复杂比较题、多跳题或覆盖多个子主题的问题则可能需要更多候选。固定 Top-K 虽然实现简单，但在真实业务里常常两边都不讨好: 简单问题拿太多噪声，复杂问题又拿不够材料。更成熟的做法是根据查询长度、查询类型、初始分数分布或历史评估结果动态决定取多少候选，再配合压缩和重排序控制最终上下文。

## 关键对比 / 易错点

- Compression 的目标是删噪，不是单纯缩字数；把关键限定条件删掉，答案往往会更不可靠。
- Parent-Child 不是重复存两份就完事，关键在于“用小块召回，用大块阅读”的分工。
- Metadata 过滤应尽量放在检索前面做，它常常比后置 reranking 更便宜。
- 固定 Top-K 简单但僵硬，动态 Top-K 更贴近真实查询差异。
- 上下文管理不是独立模块，它和 chunking、retrieval、reranking、prompt 设计是联动的。

## 学习检查点

- 为什么说 Contextual Compression 追求的是信噪比，而不是“越短越好”？
- Parent-Child 检索分别让 child 和 parent 承担什么角色？它解决了什么矛盾？
- 哪些 metadata 字段最值得在你的语料里提前标注，为什么？
- 固定 Top-K 在什么情况下会同时造成“简单题噪声多”和“复杂题证据少”这两类问题？

## 实践建议

- 盘点你的语料是否已经有时间、来源、版本、文档类型等元数据，没有的话先补索引设计，再考虑更贵的优化。
- 做一次对比实验: 同一批问题分别走“固定 Top-K”与“动态 Top-K + compression”，观察上下文长度、延迟和回答质量的变化。
