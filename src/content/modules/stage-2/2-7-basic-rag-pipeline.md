## 为什么学这个

前面几课分别讲了 Embedding、相似度、预处理、分块和向量数据库，但如果你脑子里还只是零散零件，真正动手时很容易不知道先接哪一段、问题应该从哪一层查起。基础 RAG Pipeline 的意义，就是把这些组件串成一条完整链路，让你看到“一个问题从进入系统到生成答案”到底经历了什么。

这也是 Stage 2 的收束课。学完它，你应该不只是知道名词，而是能够画出一个最小可用的文档问答助手架构，并理解每个节点出错时会把问题传导到哪里。这种全局视角，是后面做检索优化、重排、多轮对话和评估的前提。

## 核心知识

基础 RAG 的端到端流程通常是：加载文档，进行清洗和分块，把每个 chunk 转成 Embedding，存入向量库；用户提问时，再把问题也转成向量，通过 Retriever 找到最相关的若干 chunk，把这些检索结果拼进 prompt，最后交给 LLM 生成答案。你可以把它理解成一条“先准备知识，再按问题取证，最后组织回答”的流水线。前处理阶段决定知识库质量，在线检索阶段决定证据是否命中，生成阶段决定答案是否忠于证据。

在 LangChain 这类框架里，这条链路会拆成清晰组件：`DocumentLoader` 负责加载原始资料，`TextSplitter` 负责切块，`Embeddings` 负责把文本映射成向量，`VectorStore` 负责存储和搜索，`Retriever` 则提供面向查询的召回接口。理解这些组件分工的价值在于，你后面调试时能更快定位问题到底出在加载、切块、嵌入、索引还是 prompt，而不是把所有问题都归咎于“模型不行”。

Prompt Template 设计是把检索结果真正变成回答依据的关键一步。常见做法是明确告诉模型“只能基于给定上下文回答，不知道就说不知道”，并把检索片段按可读方式插入提示词中。Top-K 的选择也会影响结果：K 太小，可能漏掉关键证据；K 太大，会把不相关片段一并塞进上下文，反而干扰回答。这个参数需要结合 chunk 策略、文档类型和模型上下文预算一起调。

LCEL（LangChain Expression Language）可以帮助你用更声明式的方式把这些组件串起来，例如把“问题输入 -> 检索 -> 格式化上下文 -> Prompt -> 模型 -> 输出解析”写成可组合的链。它的价值不只是语法简洁，更在于让数据流显式可见，便于替换组件、插入调试节点和做后续扩展。对于课程项目“个人文档问答助手”，一个最小闭环应包括：导入个人文档、构建索引、输入问题、返回带出处的答案，以及在找不到证据时明确提示不确定。

## 关键对比 / 易错点

- RAG pipeline 不是“向量检索 + 大模型”这么简单，中间的清洗、切块和 prompt 注入都会显著影响最终效果。
- 检索结果放进 prompt 不是越多越好。Top-K 过大时，无关上下文会挤占预算并稀释关键信息。
- prompt 里写“仅依据上下文回答”能降低幻觉，但不能替代高质量召回；证据错了，模型仍会忠实地答错。
- LangChain 组件帮助你组织流程，但不会自动替你做正确设计。理解组件职责比会背 API 名字更重要。
- LCEL 的价值在于显式组合与可替换性，不是“更高级的写法”。如果不理解链路，它只会把问题包得更深。

## 学习检查点

- 你能否从文档导入开始，完整说出基础 RAG Pipeline 的每一个步骤及其输入输出？
- `DocumentLoader`、`TextSplitter`、`Embeddings`、`VectorStore`、`Retriever` 在 LangChain 中分别负责什么？
- Top-K 过小和过大各会带来什么问题？为什么它不能脱离 chunk 策略单独决定？
- 如果你的文档问答助手开始频繁答非所问，你会优先排查链路中的哪几个环节？为什么？

## 实践建议

- 先画出你的“个人文档问答助手”数据流图，至少标出加载、分块、嵌入、存储、检索、提示增强、生成七个节点。
- 后续编码时，为每个环节保留可观察输出，例如打印 chunk 数量、查看检索到的 Top-K 文档、记录最终 prompt，避免把整条链路当黑盒。
