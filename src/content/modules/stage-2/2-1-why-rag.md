## 为什么学这个

很多人第一次接触 LLM 时，会被它“什么都能回答”的表象迷惑，直到真正把它放进业务里才发现问题集中爆发：它不知道最近更新的制度，会把不存在的事实说得很像真的，也不了解你公司的内部知识。只靠 prompt 调整，往往治标不治本。

RAG 的价值就在这里。它不是要求模型凭记忆硬答，而是先去外部知识源里找材料，再带着材料回答问题。你要先建立这个“开卷考试”的直觉，后面学 Embedding、分块、向量库和检索策略时，才知道每个环节到底在为哪类问题服务。

## 核心知识

LLM 在真实应用里最常见的三类痛点是：知识过时、幻觉、缺乏私域知识。知识过时指模型参数里固化的是训练时见过的信息，产品规则、公司制度、API 文档一旦更新，模型并不会自动同步。幻觉则是模型在缺少依据时，仍然会按语言模式生成“看起来像答案”的内容。私域知识问题更直接，企业内部文档、客服 SOP、会议纪要、代码库细节通常不在公开训练语料里，模型天然不知道。

RAG 的核心思想是把回答流程拆成两步。第一步不是直接让模型作答，而是先根据用户问题去知识库里检索相关片段；第二步再把这些片段作为上下文注入 prompt，让模型基于证据生成答案。你可以把它想成把闭卷考试改成开卷考试，模型本身还是那个模型，但它回答时多了一本“当前可查的参考资料”。因此 RAG 的重点不是让模型变聪明，而是让它在回答前先拿到更准、更近、更私有的材料。

RAG 和微调经常被混为一谈，但它们解决的问题不同。RAG 更擅长补知识，让模型在不改参数的前提下接入最新资料和私域信息；微调更擅长改行为，比如让模型学固定输出格式、特定语气、行业任务模式。成本上，RAG 更偏工程建设，成本集中在文档处理、向量检索和在线链路；微调则需要准备训练数据、跑训练和持续维护模型版本。知识经常变化时，优先考虑 RAG；如果核心问题是模型“会不会按你想要的方式回答”，才考虑微调。

典型应用场景包括企业知识库问答、客服系统、文档问答和代码助手。它们有共同点：答案依赖外部资料，而且资料会更新。比如客服机器人要回答最新退换货规则，代码助手要参考当前仓库实现，企业知识库要基于内部制度和流程解释问题。这些都不是单靠模型记忆能稳定解决的。

## 关键对比 / 易错点

- RAG 解决的是“回答前补充证据”，不是“永久把知识写进模型参数”。文档更新后通常只需要重建索引，不需要重新训练模型。
- 微调不等于更懂知识库。没有把新知识作为外部上下文提供给模型时，微调也无法可靠记住大量持续变化的文档内容。
- 检索到了文档不代表答案就一定正确。如果召回片段不相关、过时或被截断，模型仍然会基于坏证据生成错误回答。
- RAG 也不是所有问题都要上。纯风格改写、固定分类、简单结构化抽取等任务，可能普通 prompt 或微调就够了。
- “让模型开卷”不代表可以无限塞材料。上下文太多、太杂同样会稀释关键信息，后面分块和 Top-K 就是在控制这个问题。

## 学习检查点

- 为什么说知识过时、幻觉和缺乏私域知识是 LLM 在业务落地中的三大痛点？
- 你能否用“开卷考试”解释 RAG 的基本流程，以及它相比直接提问多了哪一步？
- 如果一个团队的内部政策每周都在变，而回答风格要求并不复杂，你会优先选 RAG 还是微调？为什么？
- 企业知识库、客服系统和代码助手为什么都适合用 RAG，而不是只靠更强的通用模型？

## 实践建议

- 选一个你熟悉的业务场景，分别写出“只靠 LLM 会错在哪”和“引入 RAG 后能补什么证据”，训练自己识别 RAG 适用条件。
- 找一份经常更新的文档，例如产品说明或接口文档，尝试设计一个最小知识库问答流程，先只画出“检索前”和“检索后”的数据流。
