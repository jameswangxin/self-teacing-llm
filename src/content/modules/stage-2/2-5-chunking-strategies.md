## 为什么学这个

就算文档已经加载干净，也不能直接把整篇文档原封不动丢给检索。块太大，召回到的内容里会混入大量无关上下文；块太小，关键信息又会被切散，模型拿到的只是碎片。Chunking 分块策略本质上是在解决“检索粒度”问题，它直接影响召回率、答案精度和最终提示词成本。

很多 RAG 初学者只会照抄一个 `chunk_size=500`、`chunk_overlap=50` 的默认配置，但这通常只是在碰运气。你需要知道不同分块方法为什么存在、适合什么文档，以及 chunk 大小和重叠比例到底在权衡什么。

## 核心知识

为什么要分块？因为嵌入和检索面对的是“文本片段”而不是整库文档。若整篇文档作为一个向量，问题只和其中一小段相关时，其他无关内容会稀释语义；若把文本切得过碎，又可能让定义、条件和例外条款分散在不同块中，导致检索回来缺少关键上下文。所以 chunking 的目标不是机械切字数，而是在可召回性与语义完整性之间找到平衡。

固定大小分块是最简单的方法，按字符数或 token 数均匀切开，常配合 overlap 使用。它实现简单、速度快、容易做基线，但缺点也明显：完全不理解段落边界，可能把一句话或一个逻辑单元从中间截断。递归字符分割是在此基础上的改良，会优先按较大的结构边界切，比如章节、段落、句子，实在超长再继续向下切。它是很多通用文本场景中的默认方案，因为兼顾了简单和效果。

语义分块更进一步，不再只看字符位置，而是尝试识别“话题是否发生明显转折”。当一个段落内部语义变化很大时，语义分块能比固定切分更自然；但它实现更复杂，也更依赖额外模型或启发式判断。结构感知分块则会利用文档已有结构，例如 Markdown 标题、FAQ 问答对、API 章节、代码函数定义等，让一个 chunk 尽量对应一个完整知识单元。对结构清晰的文档，这通常比纯字符切分更稳。

Chunk size 与 overlap 是实操中的核心旋钮。块太大，单块语义混杂、召回不精准，还会增加送进 prompt 的 token 成本；块太小，相关信息可能被拆开，检索虽然“命中”了，但模型回答仍不完整。overlap 的作用是降低边界切断风险，让相邻块共享一部分内容；但 overlap 太高会制造冗余、增加索引量，并让检索结果出现大量内容重复。调优时要看文档类型、问答粒度和上下文预算，而不是追求某个通用神奇数字。

## 关键对比 / 易错点

- 分块不是越细越好。切太碎会让定义、条件和结论分散，导致模型拿到“局部正确、整体不够”的证据。
- 固定大小分块是基线，不是最佳实践。结构明显的文档通常更适合递归或结构感知分块。
- overlap 用来补边界，不是越大越安全。过高 overlap 会增加存储、检索重复和 prompt 冗余。
- 语义分块听起来更高级，但并不总比简单规则更值。文档结构本来就清晰时，复杂方案可能收益有限。
- chunk size 要和后续 Top-K、模型上下文窗口一起看，不能脱离整条 RAG 链路单独设定。

## 学习检查点

- 为什么不能直接把整篇文档作为一个向量去做检索？
- 固定大小分块、递归字符分割、语义分块和结构感知分块分别适合什么场景？
- chunk size 过大或过小，各会带来什么典型问题？
- overlap 的主要作用是什么？它为什么也可能带来副作用？

## 实践建议

- 选一篇 FAQ 或技术文档，分别用“按固定长度切”和“按标题结构切”的思路画出 chunk 边界，对比哪一种更像完整知识单元。
- 在后续实作里，至少尝试两组不同的 `chunk_size` 和 `overlap` 配置，并记录它们对召回质量和上下文成本的影响。
