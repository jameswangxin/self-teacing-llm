## 为什么学这个

如果你把 LLM 只理解成“会聊天的机器人”，后面学 Prompt、RAG、Agent 时很容易停留在堆技巧的层面。先建立正确直觉，才能知道模型为什么有时表现惊艳，有时又会一本正经地胡说八道。

这一课的目标不是学数学推导，而是形成工程判断：模型到底在做什么，训练和使用分别发生在什么阶段，不同类型模型怎么选，以及它天然不擅长什么。这个框架会决定你之后如何设计 prompt、评估结果和设置系统预期。

## 核心知识

LLM 可以先粗略理解为一个“根据前文预测下一个 token”的系统。它之所以看起来像在理解语言，是因为 Transformer 架构擅长在很大的上下文里找相关关系：一句话前面的定义、后面的补充、前几轮对话里的语气和约束，都可能影响当前这一步预测。你不需要先掌握注意力公式，但要知道模型不是在数据库里查标准答案，而是在根据训练中见过的大量模式做概率生成。

从生命周期看，LLM 常见有三个阶段。预训练是在超大规模语料上学习语言和世界模式，相当于打基础；微调是在特定任务或风格数据上继续训练，让模型更贴近某类需求；推理则是你真正调用 API 或在聊天界面提问的阶段，这时模型不会“现场学习新知识”，只是结合当前输入和已有参数生成输出。很多初学者把“这次聊天告诉模型一件事”误以为“模型被永久教会了”，这是典型误解。

模型选型还要区分开源与闭源。闭源模型通常在综合能力、服务稳定性、生态支持上更成熟，但可控性更弱、成本和供应商锁定风险更高；开源模型更适合私有部署、定制和成本优化，但要自己承担部署、推理性能和效果调优。实际工作里，不是“谁最强就用谁”，而是根据任务质量、预算、合规、延迟和是否允许本地部署综合判断。

最后必须正视能力边界。LLM 有知识截止日期，可能不知道最新事实；它会产生幻觉，也就是在不确定时仍然编造听起来合理的内容；它能做一定推理，但复杂、多步、需要高精度约束的任务仍会出错。工程上要把它视为“强大的语言接口”，而不是“永远正确的专家系统”。

## 关键对比 / 易错点

- “会生成流畅文本”不等于“真正理解事实”。流畅性是语言模式拟合得好，不代表每句话都可靠。
- 预训练、微调、推理是三个不同阶段。你在对话里补充背景，属于推理阶段的上下文注入，不是重新训练模型。
- 开源 vs 闭源不是能力高低的单选题。前者偏可控和可部署，后者偏即插即用和综合表现。
- 幻觉不是偶发现象，而是生成式模型的结构性风险。越缺少证据、越要求具体事实时越要警惕。
- “推理模型”也不是万能。遇到长链条计算、隐含条件多或任务定义模糊时，仍然可能稳定地产生错误答案。

## 学习检查点

- 你能否用“预测下一个 token”解释 LLM 为什么看起来像在理解语言？
- 预训练、微调、推理三者分别在什么时候发生，各自改变了什么？
- 如果团队要求本地部署、控制成本和自定义能力，你会更偏向开源还是闭源模型？为什么？
- 为什么知识截止日期和幻觉会同时影响模型输出的可信度？

## 实践建议

- 选一个你常用的模型，写下它的强项、弱项、是否闭源、是否支持私有部署，练习做一次选型说明。
- 用同一个事实性问题去问两个不同模型，记录它们在哪些地方会“自信但不准确”，训练自己的风险意识。
