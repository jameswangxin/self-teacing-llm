## 为什么学这个

基础 prompt 能解决“把需求说清楚”，但一旦任务变复杂，比如多步推理、长链条判断、跨阶段处理，就需要更高级的策略。很多工程效果提升，并不是换了更强模型，而是把任务拆得更合理、验证做得更明确。

进阶 Prompt Engineering 的价值，在于让你从“单次提问”转向“任务设计”。你会开始思考模型在哪一步最容易错、怎么让它更稳定、哪些任务应该拆成多个环节，而不是希望一条超级 prompt 解决所有问题。

## 核心知识

Chain of Thought（CoT）的核心思想是让模型显式展开中间推理步骤，从而在复杂任务上减少跳步出错。它并不保证绝对正确，但在数学、逻辑、多条件判断等任务里，通常比直接要答案更稳。Self-Consistency 则是在较高随机性下生成多份推理路径，再对结果做多数投票或交叉比对，本质上是用多次采样换更高鲁棒性，代价是成本和延迟都会上升。

Role Prompting 不是简单加一句“你是专家”，而是把任务所需视角、约束和评价标准写清楚。例如“你是风控分析师，需要优先识别风险并说明证据不足处”比“你是金融专家”更有效。Prompt Chaining 则适合复杂任务拆解：先抽取事实，再分类，再生成结论，每一步只做一件事。这样不仅更稳定，也更容易定位错误来自哪一环。

常见反模式包括：一个 prompt 里同时塞十个目标；把不确定事实当确定前提；要求模型既自由发散又必须格式严谨；不给验证步骤却希望结果可靠。进阶策略的重点不是堆花活，而是降低单步复杂度、增加可观察性，并在关键环节加入检查。

## 关键对比 / 易错点

- CoT 适合提升复杂推理稳定性，但不适合所有任务。简单抽取或分类问题，加入长推理反而浪费 token。
- Self-Consistency 的核心是“多次采样后汇总”，不是简单重试一次。它换来更稳结果，也带来更高成本。
- Role Prompting 有效的前提是角色能带来明确判断框架，而不是只换一个好听的身份标签。
- Prompt Chaining 关注的是任务拆分与接口设计，不是把一长段 prompt 随意拆成几段。
- 进阶技巧不能掩盖事实缺失。如果输入材料本身不够，链条再漂亮也只是在更稳定地输出不完整答案。

## 学习检查点

- 什么类型的任务更适合使用 CoT，什么类型的任务没必要使用？
- Self-Consistency 为什么能降低随机性带来的波动？它的代价是什么？
- 一个好的 Role Prompting，除了身份，还应该补充哪些信息？
- 当复杂 prompt 经常失败时，你如何判断是否应该改成 Prompt Chaining？

## 实践建议

- 选一个多步判断任务，分别尝试“直接回答”和“先列步骤再回答”，比较准确率与可解释性。
- 把一个复杂需求拆成 2 到 3 个串联 prompt，并为每一步定义输入输出格式，练习构建小型工作流。
