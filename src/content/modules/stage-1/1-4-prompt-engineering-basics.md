## 为什么学这个

Prompt Engineering 的本质不是“写咒语”，而是把任务说明书写到足够清楚，让模型更容易按照你的目标输出。初学者最常见的问题不是不会提问，而是不会定义任务边界、成功标准和输出格式。

这一课会帮你建立一套基础工作流：先安排角色分工，再决定是否给例子，然后把指令写清楚、把输出格式定死，最后用调试方法定位问题。这样你面对不稳定输出时，能有系统地改，而不是一句一句碰运气。

## 核心知识

在聊天式 API 中，system、user、assistant 三种角色承担不同职责。System prompt 用来定义全局规则、身份、风格和硬约束，例如“你是一个严谨的代码审阅助手，优先指出 bug”；user prompt 提供当前任务、输入材料和具体问题；assistant 消息则常用来承接历史回答或 few-shot 示例。理解这三者的分工后，你就不会把所有信息都胡乱塞进一条用户消息里。

Zero-shot 是不给例子，直接提要求；few-shot 是给模型若干输入输出示例，让它模仿任务模式。前者成本低、写起来快，适合简单明确任务；后者更适合风格一致、格式固定或边界微妙的任务，例如分类标准比较抽象、摘要风格有特定要求时。Few-shot 的价值不在“多给几个例子”，而在于用例子明确标准。

清晰指令通常包含四件事：任务目标、输入范围、输出格式、评价标准。比如“总结下面文章”太模糊，而“用 3 个要点总结文章，只保留结论，不要复述背景，输出 JSON”就清楚得多。结构化输出则进一步把自由文本变成易处理数据，例如 JSON、Markdown 表格、XML 标签。若结果不符合预期，调试时不要一次改很多处，而是依次检查：任务是否清楚、示例是否足够、格式是否过严、上下文里是否有冲突指令。

## 关键对比 / 易错点

- System prompt 不只是“写个身份设定”。它更适合放长期有效的规则和约束，而不是一次性任务细节。
- Few-shot 不是例子越多越好。无代表性的例子会误导模型，还会增加 token 成本。
- “写得详细”不等于“写得清楚”。冗长但没有结构的 prompt，往往比简短但明确的 prompt 更差。
- 要求 JSON 输出时，只说“请返回 JSON”通常不够，最好明确字段名、类型和缺失值规则。
- Prompt 调试失败的常见原因是一次同时改角色、任务、例子、格式，导致你无法知道到底哪一步起作用。

## 学习检查点

- System、user、assistant 三种角色分别最适合承载什么信息？
- Zero-shot 和 few-shot 的选择，应主要看任务的什么特征？
- 一个“指令不清晰”的 prompt，通常缺了哪些关键元素？
- 当模型返回的 JSON 经常不合法时，你会按什么顺序排查问题？

## 实践建议

- 把一个你常用的模糊 prompt 改写成“角色 + 任务 + 输入 + 输出格式 + 约束”的结构化版本。
- 为一个分类或抽取任务各写一版 zero-shot 和 few-shot prompt，对比准确率、稳定性和 token 成本。
