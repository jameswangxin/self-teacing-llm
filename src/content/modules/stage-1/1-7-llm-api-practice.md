## 为什么学这个

前面学的模型、token、参数和 prompt，如果不能落到一次真实 API 调用上，就还停留在概念层。真正做工程时，你需要面对消息结构、流式输出、错误处理、模型切换和成本控制，这些都不会自动发生。

这一课把 Stage 1 的知识串起来：你要把 prompt 变成程序输入，把模型参数变成接口参数，把上下文管理变成代码逻辑，并最终能规划一个可用的命令行聊天助手。这是从“会提问”走向“会构建 AI 应用”的第一步。

## 核心知识

OpenAI-compatible 聊天接口通常围绕一个消息数组展开，每条消息带有 `role` 和 `content`，再配合模型名、temperature、max tokens 等参数。你要理解接口层和模型层的边界：接口负责把输入组织成服务商接受的格式，模型负责生成结果。换服务商时，很多概念相同，但 base URL、鉴权方式、字段细节和模型命名可能不同，所以把调用封装成统一客户端会更容易维护。

流式响应的价值在于降低等待焦虑。服务端会把生成内容分块推送给客户端，你的程序则要持续读取事件流、逐步渲染文本，并在结束时正确收尾。这要求你处理半截 token、终止事件和异常中断。多模型接入时，还要留意上下文长度、速率限制、价格和是否支持某些参数，因为“兼容 OpenAI”通常只是大体兼容，不代表所有细节完全一样。

错误处理是 API 实战的基本功。常见问题包括速率限制、token 超限、网络超时和临时服务错误。应对方式通常是：先区分可重试和不可重试错误；对可重试错误使用指数退避；为请求设置合理超时；在 CLI 项目里保存对话历史并允许裁剪上下文。一个支持多模型切换的命令行聊天助手，最小闭环应该包括：选择模型、发送消息、流式展示回复、保留历史、处理失败重试。

## 关键对比 / 易错点

- “OpenAI-compatible” 不等于“完全一致”。你不能假设所有服务商都支持相同参数、相同错误码和相同流式事件格式。
- 流式输出解决的是交互体验，不是成本问题。token 还是照常计费，失败中断也可能已经产生费用。
- 重试机制不是对所有错误都生效。鉴权失败、参数错误这类问题，重试只会重复失败。
- 保存完整对话历史虽然简单，但很快会碰到 token 上限和成本问题，需要摘要或裁剪策略。
- 多模型切换不只是改个模型名，还要考虑默认参数、能力差异和回退策略。

## 学习检查点

- 聊天接口里的消息数组，为什么比单一字符串 prompt 更适合真实应用？
- 流式响应下，客户端为什么需要按块处理数据而不是等完整文本一次返回？
- 哪些错误适合重试，哪些错误应该立刻暴露给用户修正？
- 如果你要做多模型切换的 CLI，最小可用版本需要哪些核心能力？

## 实践建议

- 写一个最小脚本，调用一个 OpenAI-compatible 聊天接口，支持传入模型名、temperature 和用户消息。
- 在此基础上加入流式输出和指数退避重试，再记录不同模型在速度、风格和成本上的差异，作为后续 CLI 聊天助手的设计输入。
