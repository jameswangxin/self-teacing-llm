## 为什么学这个

很多“模型太笨了”的问题，根源并不是模型能力，而是你没有意识到 token 和上下文窗口在背后限制一切。提示词太长、历史对话堆太多、输出长度没预算，这些都会直接影响效果、速度和成本。

理解 token 之后，你会更像一个工程师而不是使用者：能估算一次调用为什么贵、为什么截断、为什么长文档问答丢信息，也能更合理地设计 prompt、切分内容和安排上下文。

## 核心知识

Token 是模型处理文本时的内部切分单位，不等于字、不等于词，也不等于字符。英文里一个常见单词可能是一个或几个 token，中文里一个汉字有时就是一个 token，有时会和相邻字组合，因此“1000 字中文”和“1000 个英文单词”对模型来说，消耗并不相同。代码、标点、JSON 键名也都会占 token，这就是为什么结构化输出虽然好处理，但也会增加成本。

上下文窗口指模型一次能“看到”的总 token 数，通常包括 system prompt、用户输入、历史对话、检索内容以及模型输出空间。4K 更适合短问答和简单指令，32K 可以容纳较长文档片段和多轮对话，128K 甚至 1M 能处理更长材料，但“能塞进去”不等于“都能被同等重视”。长上下文里，中间信息更容易被忽略，尤其当任务目标不清楚、无关内容太多时，这就是常说的注意力衰减。

计费通常按输入 token 和输出 token 分别收费。输入越长、输出越多，调用越贵；如果你把很长的聊天历史原样回传，每次成本都会继续上升。工程上要学会做预算：哪些上下文必须保留，哪些可以摘要，哪些应该重新检索而不是一直携带。

## 关键对比 / 易错点

- Token 不是“字符数统计”。同样长度的中文、英文、代码和表格，token 消耗可能差很多。
- 上下文窗口是“输入加输出”的总预算，不是只算用户提问。给模型留不出输出空间，结果就会截断。
- 128K 或 1M 不代表长文任务一定更好。无关信息堆得越多，模型越容易抓不住真正重点。
- 输入 token 和输出 token 都会计费。只盯着 prompt 长度，却忽略冗长回答，也会导致成本失控。
- 很多人把“历史全带上”当成多轮对话最佳实践，但当历史已经偏题或重复时，实际上是在污染上下文。

## 学习检查点

- 为什么“一个汉字等于一个 token”只是近似说法，而不是严格规则？
- 在一次 32K 上下文调用里，为什么你仍然需要为输出预留 token 预算？
- 如果你做长文档问答，什么情况下应该摘要历史，什么情况下应该重新检索片段？
- 为什么长上下文会出现“中间内容被忽略”的现象，这对 prompt 设计有什么影响？

## 实践建议

- 拿一段中文介绍、一段英文文章和一段 JSON 样例，分别估算哪个最费 token，再用实际 tokenizer 工具验证。
- 设计一个长文档问答 prompt，对比“全文直接塞入”和“先切块再挑重点输入”两种方案的效果与成本。
