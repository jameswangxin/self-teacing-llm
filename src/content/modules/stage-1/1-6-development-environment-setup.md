## 为什么学这个

很多 AI 项目不是死在模型效果，而是死在环境混乱：今天能跑、明天不能跑，依赖冲突、Notebook 到处散落、API Key 写进代码仓库。开发环境搭建看起来不“聪明”，但它决定了你能不能持续迭代。

这一课要解决的是工程基本盘。你需要一个可复现的 Python 环境、清晰的目录结构、能管理依赖的方式，以及安全处理 API Key 的习惯。这样后面做 API 实验、RAG 和 Agent 项目时，才不会一边学概念一边被环境问题拖住。

## 核心知识

Python 环境管理最常见的两条线是 `venv` 和 `uv`。`venv` 是 Python 自带的基础方案，足够通用，适合理解虚拟环境是什么；`uv` 更像现代化工具链，兼顾创建环境、安装依赖和更快的包管理体验。无论用哪种方案，核心原则都一样：项目依赖要和系统 Python 隔离，避免不同项目之间互相污染。

项目结构方面，AI 项目至少要区分几个区域：源码目录、实验目录、数据目录、脚本目录和配置文件。比如 `src/` 放正式代码，`notebooks/` 放探索性实验，`scripts/` 放一次性或自动化脚本，`data/` 放样例或处理中间产物。这样做的目的不是“看起来专业”，而是让你知道哪些内容可以重构成产品代码，哪些只是实验草稿。

依赖管理上，`pyproject.toml` 更适合现代项目，信息集中、工具兼容性好；`requirements.txt` 则适合快速记录一组可安装依赖，常用于部署或冻结版本。Notebook 适合做交互式探索，但不适合承载全部业务逻辑，验证过的代码要尽早迁移回模块文件。API Key 管理则必须放进环境变量或 `.env` 文件，并通过 `python-dotenv` 等工具加载，绝不要硬编码到脚本或提交到仓库。

## 关键对比 / 易错点

- `uv` 和 `venv` 的差异主要在工具体验和效率，不在“是否更专业”。关键是你有没有真正隔离环境。
- Notebook 适合实验，不适合长期堆积核心逻辑。很多项目后期难维护，就是因为所有代码都留在单个 `.ipynb` 里。
- `pyproject.toml` 管理的是项目层依赖元数据，`requirements.txt` 更像安装清单，两者用途相关但不完全等同。
- `.env` 文件方便，但前提是加入忽略规则并避免泄露。把密钥放在源码里是高风险错误。
- 目录结构不是越多越好。过度设计一堆文件夹，只会让早期项目更难找东西。

## 学习检查点

- 为什么同一个项目应该使用独立虚拟环境，而不是直接装到全局 Python？
- `pyproject.toml` 和 `requirements.txt` 各自更适合哪些场景？
- 你会如何区分应该留在 Notebook 的代码和应该迁移到 `src/` 的代码？
- 为什么 `.env` 能提高便利性，但仍然需要配合忽略规则和最小暴露原则？

## 实践建议

- 用 `uv` 或 `python -m venv` 新建一个干净环境，安装一个最小依赖集，并记录完整启动步骤。
- 为你的下一个 AI 小项目先搭好 `src/`、`notebooks/`、`scripts/` 和 `.env.example`，再开始写第一行业务代码。
