## 为什么学这个

学会 LangGraph 基础概念后，你已经能把简单流程跑起来，但“能跑”离真实工程还差很远。复杂 Agent 系统真正难的地方，是状态会跨多轮存在，路径会动态分支，某些节点要循环重试，某些结果要等待人工批准，某次崩溃后还希望能从中间恢复。如果没有这些能力，图只是把流程画得更清楚，却还没有真正提升系统韧性。

这一课关注的，就是 LangGraph 为什么适合做长期、复杂、可恢复的 Agent 工作流。你要把重点从“如何定义图”转到“如何让图在不稳定现实里活下来”: 怎么循环、怎么复用子图、怎么落检查点、怎么流式观察、怎么对失败做重试而不是直接崩掉。

## 核心知识

Conditional Edges 是复杂图的入口。基础条件分支只是在两个路径中做选择，而高级用法强调“用状态驱动流程重配置”。例如，某个评估节点可以根据检索质量、模型置信度或外部 API 结果，把请求路由到继续搜索、降级回答、进入人工审批或结束输出。关键不在于分支本身，而在于决策依据已经落进可观察状态，而不是散落在黑盒 prompt 里。

Cycles 让图具备“试错直到满足条件”的能力。常见例子包括: 搜索结果不足就改写查询再检索，代码执行失败就修复后重跑，草稿回答经评估不过关就回到补证据节点。循环本身不是问题，失控循环才是问题，所以工程上通常会同时设计退出条件、最大轮次和错误标记。图能表达循环，不代表你应该无限循环。

Subgraphs 的价值是把复杂流程模块化。你可以把“做一次检索评估”“执行一次人工审批”“运行一个研究子任务”封装成子图，在主图里像节点一样调用。这样既降低主流程复杂度，也让某些可复用能力具备清晰边界。多 Agent 系统里，子图尤其有用，因为单个 Agent 的内部流程完全可以被包成一个独立子图，再由上层图协调多个子图之间的顺序与状态交换。

Checkpointing 解决的是“流程跑到一半怎么办”。长流程 Agent 不可避免会遇到服务中断、人工等待、进程重启或需要审计回放的情况。检查点把状态和执行位置持久化下来，让流程可以恢复，而不是每次从头开始。它不仅提升稳定性，也让 interrupt/resume、人工审批和长周期任务真正可行。配合 Streaming，你还能在执行时持续看到节点事件、状态更新或中间产物，让调试和用户体验都更透明。

错误处理与重试是高级特性的落地重点。并不是所有失败都该重试: 参数校验失败通常应快速报错，临时网络异常更适合重试，幂等写操作与非幂等外部调用的策略也不同。图工作流必须显式区分“可恢复失败”和“不可恢复失败”，并在节点设计里考虑超时、回退、补偿和幂等性。否则重试机制只会把一次错误放大成多次错误。

## 关键对比 / 易错点

- 条件边的重点不是“多几个 if/else”，而是让状态驱动流程选择，便于观察与恢复。
- 循环很强，但没有退出条件的循环只是在把 Agent 的失控行为制度化。
- 子图不是为了追求抽象优雅，而是为了把复杂流程切成可复用、可理解的边界。
- Checkpointing 不是纯性能优化，它首先解决恢复、审计和人工介入问题。
- 重试并不总是安全；如果节点有副作用，必须先想清楚幂等与补偿。

## 学习检查点

- 在什么场景下，你会优先使用条件边而不是让模型在 prompt 里自己决定下一步？
- 为什么循环节点必须配合退出条件、轮次上限或失败标记一起设计？
- 子图在多 Agent 或复杂流程中能解决什么具体问题？
- 一次节点失败时，你如何判断它适合重试、适合回退，还是应该直接中断？

## 实践建议

- 把一个“检索不够好就重写查询再查一次”的流程画成带循环和条件边的图，并为它明确最大重试次数。
- 给你的图状态加上执行轮次、错误类型和人工审批标记，再思考这些字段如何帮助 Checkpointing 与恢复。
