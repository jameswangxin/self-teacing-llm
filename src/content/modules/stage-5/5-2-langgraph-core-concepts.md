## 为什么学这个

很多人第一次看 LangGraph，会觉得它只是把“链”换成了“图”，概念上似乎并不难。但真正卡住初学者的，不是 Node 和 Edge 这些名词，而是为什么系统突然开始强调 State、为什么定义完图还要 `compile()`、为什么执行时既可以 `invoke()` 又可以 `stream()`。如果这些基础概念没有串起来，后面学高级特性时会始终像在背 API。

这一课的目标，是帮你建立 LangGraph 的最小心智模型: 图是怎么定义的，状态是怎么流动的，节点如何读写状态，边如何决定去向，图如何从静态结构变成可执行对象。只要这一层稳住，后续的循环、子图、检查点和人工中断都会自然得多。

## 核心知识

LangGraph 的第一层是有向图。Node 代表一个可执行步骤，它可以是模型调用、工具函数、检索器、校验器，甚至人工审批。Edge 代表控制流，说明一个节点完成后默认该去哪里。之所以强调“有向”，是因为流程不是任意跳跃，而是要明确下一步的可能方向。你可以把 Node 看成“做什么”，把 Edge 看成“做完以后去哪里”。

State 是 LangGraph 最关键、也最容易被低估的概念。在线性 Agent 里，很多上下文靠消息拼接和隐藏状态传递；在 LangGraph 里，状态被提升为一等公民。节点输入通常是当前状态，节点输出通常是对状态的更新。这样做的好处是，流程里的中间结果、错误标记、检索证据、审批结论都可以显式存放在统一状态里，而不是散落在局部变量和 prompt 模板中。状态越清晰，图越容易调试和恢复。

Node 和 Edge 都有不同类型。普通节点是顺序执行的一步，条件节点则会根据状态返回不同分支；普通边表示固定流向，条件边表示“根据当前判断选择下一跳”。这套机制的核心价值，是把“动态决策”从模型内部独白转成系统级控制。比如你可以根据 `retrieval_score` 决定继续检索还是直接回答，而不是完全依赖模型自发判断。

LangGraph 从定义到执行一般经历两步。先声明节点、边和状态模式，构造出一个图；再通过 `compile()` 把这份图定义编译成可执行应用。编译后的对象才是运行入口。随后你可以用 `invoke()` 一次性跑完整个图，也可以用 `stream()` 边执行边观察中间事件。前者更适合后端同步调用，后者更适合调试、可视化和前端实时反馈。理解 `compile() -> invoke()/stream()`，本质上是在理解“声明式图定义”和“运行时执行”之间的分层。

## 关键对比 / 易错点

- Node 不等于模型调用；任何可独立执行的一步都可以是节点，模型只是其中一种。
- State 不是附属品，而是 LangGraph 的中心。状态设计混乱时，图再漂亮也难维护。
- `compile()` 不是可有可无的样板代码，它是在把图定义转成可执行运行时。
- 条件分支不一定要交给模型决定，很多时候更稳的做法是根据显式状态走条件边。
- `invoke()` 强调拿最终结果，`stream()` 强调持续看到中间过程和事件。

## 学习检查点

- 在你的理解里，Node、Edge、State 三者分别承担什么职责？
- 为什么说 LangGraph 把状态从“隐式上下文”提升成了“显式数据结构”？
- `compile()` 在整个执行模型里扮演什么角色？
- 如果你要根据检索分数决定下一步路径，这个逻辑更适合写在节点里、状态里还是条件边里？为什么？

## 实践建议

- 用纸或白板画一个最小 LangGraph: 一个检索节点、一个判断节点、一个回答节点，再给它设计一个最小状态对象。
- 写示例代码时先只做 `invoke()` 版本，确认状态流清楚之后，再补 `stream()` 观察每一步事件输出。
