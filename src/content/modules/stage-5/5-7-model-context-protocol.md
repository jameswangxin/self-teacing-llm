## 为什么学这个

Function Calling 让模型学会了“按 schema 调工具”，但当工具越来越多、来源越来越杂、上下文不只包含工具还包含资源和提示模板时，单一厂商的函数调用接口就开始显得不够通用。你会碰到重复定义工具、接入方式不统一、动态发现能力弱、客户端与服务端耦合过深的问题。

MCP 的重要性，在于它试图把“模型应用如何接外部能力”标准化。学这一课，不是为了死记协议细节，而是为了建立一个更高层的视角: 工具、资源、提示词如何以统一方式被发现和使用，Host、Client、Server 各自负责什么，以及它和 Function Calling 到底是替代关系还是互补关系。

## 核心知识

MCP 可以理解成面向模型应用的上下文接入协议。它关心的不只是某个函数能不能被调用，而是一个宿主应用如何与外部能力提供方建立统一连接，并动态获取可用能力。这里通常有三个角色。Host 是承载用户体验和 Agent 逻辑的应用，例如 IDE、桌面客户端或 Agent 平台；Client 是 Host 内部负责发起 MCP 通信的一侧；Server 则暴露能力，告诉外部“我这里有哪些工具、资源和提示可用”。

Tools、Resources、Prompts 是 MCP 里三类重要对象。Tools 对应可执行能力，例如搜索、写文件、调 API；Resources 更像可读取上下文，例如文档、数据库视图、配置文件、知识条目；Prompts 则提供可复用的提示模板或交互脚手架。把这三类对象统一进协议的意义在于，模型应用不再只是在“调用一个函数”，而是在接入一个完整能力源。

动态发现是 MCP 相比硬编码工具接入的一大优势。传统做法里，客户端常常要预先知道所有工具定义，并把 schema 写死在代码里。MCP 更强调运行时发现: Host 连接到 Server 后，可以查询它支持哪些工具、资源和提示，再按统一方式展示和使用。这对工具生态扩展很重要，因为它降低了接入成本，也减少了每增加一个工具就要改一遍宿主代码的问题。

MCP 与 Function Calling 的关系，最容易被误解。Function Calling 更像模型与调用层之间的交互机制: 模型根据已知 schema 产出结构化调用意图。MCP 更像能力暴露与发现协议: 它解决“这些能力如何被统一描述、发现和访问”。在很多系统里，两者完全可以配合使用。你可以用 MCP 把工具和资源接进来，再由上层 Agent 通过 function calling 或其他决策机制选择具体调用哪个能力。

## 关键对比 / 易错点

- MCP 不只是“更多工具”，它的重点是统一接入、发现和上下文表达方式。
- Host、Client、Server 不是三个孤立程序名词，而是协议交互中的职责分工。
- Resources 和 Prompts 的存在说明 MCP 管的不只是可执行函数。
- Function Calling 关注模型如何提出调用意图，MCP 关注能力如何被标准化提供出来。
- 引入 MCP 后也不意味着自动获得好用的 Agent；编排、权限控制和评估仍然要自己做。

## 学习检查点

- 在 MCP 体系里，Host、Client、Server 三者各自负责什么？
- 为什么说 MCP 把“工具接入问题”提升成了“能力发现与上下文接入问题”？
- Tools、Resources、Prompts 三类对象分别适合承载什么能力？
- MCP 与 Function Calling 可以如何配合，而不是互相替代？

## 实践建议

- 挑一个你熟悉的工具接入场景，尝试改用“能力发现”的视角描述它，看看哪些信息适合归入 Tools、Resources 或 Prompts。
- 在设计 Agent 平台时，把“工具 schema 存在哪里、如何发现、如何授权”单独拿出来思考，不要全部混进 prompt 或业务代码里。
