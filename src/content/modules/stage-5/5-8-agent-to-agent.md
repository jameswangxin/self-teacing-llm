## 为什么学这个

当系统从“一个 Agent 调很多工具”继续演进到“多个 Agent 来自不同团队、不同平台、不同运行环境，还要一起完成任务”时，问题就不再只是工作流怎么编排。你还需要一种方式，让 Agent 彼此知道对方能做什么、如何发起任务、如何交换中间消息、如何接收交付物。否则多 Agent 协作只能停留在同一代码仓里的私有协议。

A2A 的意义，就在于把 Agent 之间的协作接口标准化。学这一课，是为了把“多 Agent”从单一框架内的实现技巧，提升到跨系统协作的协议视角。同时你也要看清它和 MCP 的关系: 一个解决 Agent 之间怎么合作，一个解决 Agent 怎么接能力源，二者结合起来才更接近真实世界的多步骤 Agentic 工作流。

## 核心知识

A2A 可以理解成面向 Agent 协作的协议层。它关注的不是底层模型 API，而是一个 Agent 如何对外描述自己、如何接收任务、如何发送消息、如何返回结果。这里几个关键对象很重要。Agent Card 用来声明能力、接口、限制和联系方式，类似 Agent 的“服务说明书”； Task 表示一个待完成工作单元； Message 是任务过程中的通信载体； Artifact 则是结构化产物，例如报告、表格、代码片段或引用材料。把这些对象标准化后，不同 Agent 才有机会跨系统协作。

MCP 与 A2A 解决的问题层次不同。MCP 更偏“Agent 如何连接工具、资源和提示”，是能力接入协议；A2A 更偏“Agent 如何和另一个 Agent 协作完成工作”，是协作协议。一个研究 Agent 可以通过 MCP 访问搜索和文档资源，再通过 A2A 把任务交给写作 Agent 或审稿 Agent。这样系统结构会更清晰: 工具接入和角色协作不再混在一套私有接口里。

A2A 的工作流直觉可以这样理解: 先通过 Agent Card 发现合适协作者，再创建 Task，随后围绕该任务交换 Message，最后沉淀一个或多个 Artifact 作为交付物。这里最关键的工程价值是，任务状态和产物边界被显式定义出来。你不再只是在聊天消息里“顺便”交代工作，而是在管理一个可追踪、可审计、可交付的协作对象。

把 A2A 与 MCP 组合起来，就能更自然地设计多步骤 Agentic RAG 项目。比如一个 Orchestrator Agent 收到用户请求后，先通过 MCP 调用检索与外部搜索，再把“证据整理任务”通过 A2A 发给 Research Agent，把“报告生成任务”发给 Writing Agent，把“风险审查任务”发给 Review Agent，最后汇总 Artifact 输出最终报告。此时 LangGraph 仍可作为本地编排层，把协议交互嵌入节点或子图中。也就是说，LangGraph、MCP、A2A 并不冲突，而是分别负责图编排、能力接入和跨 Agent 协作。

## 关键对比 / 易错点

- A2A 不是“多 Agent 的另一个框架 API”，它的核心目标是标准化 Agent 间协作接口。
- Agent Card 不是宣传页，而是能力、限制和交互方式的正式描述。
- Message 传的是过程沟通，Artifact 承担的是可交付结果；两者不要混成一类。
- MCP 与 A2A 不在同一层次，前者更像能力接入协议，后者更像协作协议。
- 多 Agent 项目里，如果任务边界和产物边界不清，协议再标准也会协作混乱。

## 学习检查点

- Agent Card、Task、Message、Artifact 四个对象分别解决什么问题？
- 为什么说 A2A 和 MCP 是互补关系，而不是二选一？
- 在一个多步骤 Agentic RAG 项目里，哪些部分更适合用 MCP，哪些部分更适合用 A2A？
- 如果两个 Agent 能互发消息，但没有明确任务与产物定义，会出现什么工程问题？

## 实践建议

- 为你设想中的多 Agent 项目先写出三张最小 Agent Card，明确每个角色的能力边界、输入输出和限制条件。
- 设计一个“检索证据 -> 写作草稿 -> 人工审批 -> 最终报告”的流程，分别标注哪些节点是 LangGraph 编排、哪些调用 MCP、哪些通过 A2A 协作。
