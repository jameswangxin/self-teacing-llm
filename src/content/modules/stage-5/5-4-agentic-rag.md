## 为什么学这个

做过基础 RAG 和进阶优化之后，你会发现很多问题并不是“检索器不够强”这么简单。有些问题需要先判断资料是否可信，有些问题需要换检索策略，有些问题问法太模糊，需要先重写查询甚至拆子问题。此时如果系统还是固定执行“检索一次 -> 拼上下文 -> 生成答案”，就会越来越像把所有复杂度都压给 prompt。

Agentic RAG 的出现，就是为了让 RAG 不再只是静态流水线，而是能根据当前证据质量动态调整路径。学这一课，不是为了把所有 RAG 都做成复杂 Agent，而是为了建立判断框架: 简单 RAG 什么时候还够用，什么时候必须让系统加入评估、反思、重试和多步决策。

## 核心知识

简单 RAG 的优势是链路短、成本低、故障点少。对于事实型问答、知识库查找、范围清晰的问题，它通常已经足够好。问题在于，一旦进入多跳问题、证据噪声大、数据质量不稳定或用户问题表达模糊的场景，单次检索就可能既拿不全信息，也无法判断拿回来的内容值不值得信。此时系统需要的不只是“再检索一次”，而是“评估当前检索结果，决定下一步该做什么”。

Self-RAG 可以理解为“让系统对自己的检索与回答过程做反思”。它会在回答前后加入自评信号，比如判断是否需要检索、当前证据是否足够、答案是否忠于证据。核心价值不是神奇地提高能力，而是让系统多了一层显式自检逻辑。CRAG 更强调 Corrective 思路: 当检索结果质量不佳时，不是硬着头皮回答，而是触发纠错动作，例如过滤低质量片段、补充外部搜索、改写查询或切换检索源。Adaptive RAG 则强调动态选择策略，针对不同问题复杂度和数据条件，在“直接回答、简单检索、复杂检索、多轮决策”之间做路由。

RAG 与 Agent 的融合，本质上是在把检索系统从固定管道升级成决策闭环。图里常见节点包括: 查询理解、检索、证据评估、查询改写、补充搜索、答案生成、答案校验。关键是这些节点不必每次都执行，而是根据状态被有条件地调度。这也是 Agentic RAG 成本更高的原因: 它换来了更高的适应性、可解释性和复杂问题处理能力，但代价是更高的延迟、更多失败点和更复杂的评估要求。

判断何时升级到 Agentic RAG，可以用一个简单框架。先看失败症状是否稳定复现: 如果只是少量尾部案例，先做 chunking、混合检索、reranking、query rewriting 等 Stage 3 优化；如果系统常常需要“先判断再决定怎么检索”，说明已经接近 Agentic RAG 区间。再看任务是否需要多源信息融合、动态策略切换、检索结果质检或外部工具辅助。只有当这些需求持续存在时，复杂闭环才值得。

## 关键对比 / 易错点

- Agentic RAG 不是“更高级的默认答案”，很多场景下简单 RAG 更稳、更快、更易评估。
- Self-RAG、CRAG、Adaptive RAG 关注点不同，不要把它们都理解成“多加一次检索”。
- 如果 Stage 3 的基础优化还没做好，直接上 Agentic RAG 往往只是在复杂系统里放大原有缺陷。
- Agentic RAG 的价值在于动态决策，不在于把流程拉长；无意义的额外步骤只会增加成本。
- 评估 Agentic RAG 时，除了答案质量，还要看决策路径是否合理、是否存在无效循环。

## 学习检查点

- 你如何判断一个 RAG 问题更适合先做 Stage 3 优化，而不是直接上 Agentic RAG？
- Self-RAG、CRAG 和 Adaptive RAG 分别更擅长补哪一类短板？
- 为什么说 Agentic RAG 的核心不是“更多检索”，而是“更好的决策”？
- 如果一个问题只需要从稳定知识库里查一个精确答案，为什么简单 RAG 往往更优？

## 实践建议

- 选 5 个你现有 RAG 系统经常失败的问题，逐个判断它们属于召回不足、证据不可信、问法模糊还是需要多步推理，再决定是否值得引入 Agentic 流程。
- 先实现一个最小闭环版本: 检索 -> 质量评估 -> 必要时改写查询并重试，不要一开始就把所有 Self-RAG 和 CRAG 机制全塞进去。
