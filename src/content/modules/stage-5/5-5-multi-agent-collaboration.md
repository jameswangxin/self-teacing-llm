## 为什么学这个

当单个 Agent 的职责开始同时覆盖搜索、规划、写作、校验、工具调用和审批协调时，系统很容易变成一个“什么都能做一点、但每一步都不够稳”的超级节点。把复杂任务拆给多个专职 Agent，看起来会增加系统复杂度，但它也带来了新的工程收益: 职责更清楚、提示词更聚焦、失败定位更容易、不同角色还能独立迭代。

不过，多 Agent 并不是“把一个 Agent 复制成多个”这么简单。真正难的是协作模式怎么选、状态怎么共享、冲突怎么解决、谁来做最终决策。这一课要帮你分清不同协作拓扑的代价与收益，并理解为什么 LangGraph 很适合把多 Agent 系统编排成可观察的流程。

## 核心知识

Supervisor 模式是最常见的多 Agent 入口。一个上层协调者负责理解目标、拆分任务、调用下游专职 Agent，并整合结果返回用户。这种模式的优点是控制集中、行为容易约束，缺点是 supervisor 容易成为瓶颈，且上层提示词会越来越复杂。Peer-to-Peer 模式则更强调 Agent 之间直接协商，适合角色相对对等、需要互相交换信息的场景，但调试和冲突治理会更难。Hierarchical 模式介于两者之间，通过多层管理结构把复杂任务进一步分层。Sequential 模式最简单，多个 Agent 按固定顺序接力，适合流程清晰但单个节点职责不同的任务。

用 LangGraph 实现多 Agent 时，一个实用思路是把“一个 Agent”视为一个节点或一个子图。简单角色可以直接作为节点存在，例如研究员节点、评审员节点、总结员节点；复杂角色则封装成子图，内部再包含自己的检索、反思、重试和工具调用逻辑。这样主图只关心角色间如何协作，而角色内部如何工作则由各自子图维护。状态设计上通常要区分共享状态与角色私有状态，避免每个 Agent 都在同一个大对象上随意写入。

多 Agent 成本主要来自协调。任务拆得太碎，协调开销会超过收益；共享状态太宽，角色之间容易互相污染；角色边界不清，多个 Agent 可能重复劳动或给出互相矛盾的结论。因此多 Agent 设计首先是组织问题，其次才是模型问题。你应该先回答“为什么这个职责需要独立 Agent”，再决定是否创建新角色。

CrewAI 和 AutoGen 常被拿来和 LangGraph 一起讨论，但它们的设计哲学不完全相同。CrewAI 更强调角色分工与团队协作抽象，让你较快搭出“谁负责什么”的组织结构； AutoGen 更强调对话式多 Agent 交互，把协商过程建模成 Agent 之间的消息往返。LangGraph 则更强调可控编排和状态流，适合在你已经明确流程边界后做稳定落地。它们可以互相借鉴思路，但你要知道自己是想要“高层协作抽象”，还是“底层执行可控性”。

## 关键对比 / 易错点

- 多 Agent 的核心收益是职责分离，不是单纯“多几个模型一起想”。
- Supervisor 模式最容易落地，但别把所有复杂度又堆回一个超级协调者。
- Peer-to-Peer 更灵活，也更难调试；没有清晰协议时，Agent 间对话很容易失控。
- LangGraph 实现多 Agent 时，最容易出问题的通常不是节点调用，而是共享状态边界设计。
- 不是每个子任务都值得独立成 Agent；如果没有独立职责和独立策略，拆分只会增加协调成本。

## 学习检查点

- Supervisor、Peer-to-Peer、Hierarchical、Sequential 四种模式各自更适合什么任务结构？
- 为什么说多 Agent 设计首先是组织边界设计，而不是模型数量设计？
- 在 LangGraph 里，把一个角色做成普通节点和做成子图，分别意味着什么？
- CrewAI、AutoGen、LangGraph 在多 Agent 设计上的关注点有什么不同？

## 实践建议

- 从一个单 Agent 研究助理出发，尝试只拆出两个角色: 研究员和评审员，先验证职责分离是否真的带来收益。
- 设计多 Agent 状态时，明确列出哪些字段允许共享、哪些字段只能由特定角色写入，避免隐式耦合。
