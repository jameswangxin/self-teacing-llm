## 为什么学这个

如果你已经用过 LangChain Agent，很快会遇到一个拐点: 简单问答和单工具调用还能靠线性循环撑住，但一旦任务开始有分支、回溯、人工审批、失败重试和多角色协作，系统行为就会变得越来越难解释。你会发现问题不再只是 prompt 写得好不好，而是流程本身没有被清晰表达。

LangGraph 的价值，在于它把原本藏在执行器里的流程控制显式化。你不再把 Agent 理解成“模型不断自己想下一步”，而是把它看成一个有状态、有节点、有路径的工作流系统。学这一课，是为了把 Agent 开发从“魔法循环”升级为“可设计、可观测、可恢复的编排工程”。

## 核心知识

传统 LangChain Agent 的强项，是让你快速得到一个能跑的 think-act-observe 闭环。它适合原型验证，也适合工具数量少、流程相对单一的任务。但它的默认控制流往往偏线性: 模型决定要不要调工具，执行器接管循环，再由停止条件结束。这种方式对初学者很友好，却不擅长表达“如果检索质量不足就重试，否则进入审核”“某一步失败后回到指定节点”“先走 A 支线再合并到 B”这类复杂流程。

LangGraph 的设计哲学是把 AI 工作流建模成图。图里最核心的变化，不是换了几个 API，而是把系统结构显式写出来: 哪些步骤是节点，状态如何在步骤间传递，下一步由哪条边决定。这样一来，复杂流程不再藏在提示词和 if/else 的混合逻辑里，而是拥有了独立的编排层。你可以把模型调用、工具执行、人工审批、结果校验都当成图上的节点，再用边表达顺序、条件和循环。

LangChain 和 LangGraph 不是互相替代关系，更像不同抽象层。LangChain 更偏“组件层”和“应用开发层”，帮你定义模型、工具、检索器、prompt、链路；LangGraph 更偏“工作流编排层”，负责把这些组件组织成稳定执行图。实际工程里常见做法是: 用 LangChain 提供模型和工具接口，用 LangGraph 管流程、状态和恢复。你可以把 LangChain 视为积木，把 LangGraph 视为拼装积木的蓝图。

真正的工程判断在于何时需要上图。如果任务只是固定步骤的 RAG、一次性工具调用，继续使用简单链路通常更稳、更便宜。只有当你明确需要分支、回路、状态持久化、审计或多 Agent 协作时，LangGraph 的额外复杂度才值得付出。图编排不是“更高级就应该默认使用”，而是为了处理线性 Agent 已经描述不清的问题。

## 关键对比 / 易错点

- LangChain Agent 的问题不只是“不够强”，而是它默认抽象更适合简单闭环，不适合表达复杂工作流。
- LangGraph 不是“另一个 Agent 框架”那么简单，它更重要的价值是把控制流和状态流显式化。
- 不是所有 Agent 都要迁移到图编排；如果没有复杂分支与恢复需求，简单链路通常更可靠。
- 把 LangGraph 当成“更多 API”会学偏，真正要理解的是工作流建模方式发生了变化。
- LangChain 和 LangGraph 经常一起用，不是二选一关系。

## 学习检查点

- 你能否说明 LangChain Agent 在什么类型的任务里最容易暴露线性执行的限制？
- 如果一个流程要求“检索失败时改写问题并重试，成功后再进入人工审批”，为什么图模型比单循环更自然？
- LangChain 和 LangGraph 分别更擅长解决哪一层问题？
- 什么情况下继续用简单链路，比直接上 LangGraph 更合理？

## 实践建议

- 选一个你已经做过的 Agent Demo，把流程画成节点和边，标出哪些地方其实已经出现分支、循环或人工判断需求。
- 把现有 Agent 的执行日志倒推成流程图，观察哪些控制逻辑目前藏在 prompt、代码分支或执行器默认行为里。
