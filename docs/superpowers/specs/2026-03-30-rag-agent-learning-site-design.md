# RAG 与 Agent 学习网站设计文档

日期：2026-03-30

## 1. 背景

当前项目目录下只有一份课程纲要文档 [GUIDE.md](/Users/wangxin/Documents/llm-learn/self-teacing-llm/GUIDE.md)。这份文档定义了一条围绕 `RAG + Agent` 的系统化学习路线，覆盖 6 个阶段、42 个模块，并包含阶段目标、知识点拆分、实战项目与学习方法论。

目标不是把 `GUIDE.md` 直接原样搬成网页，而是围绕这份纲要构建一个适合本地自学的轻量学习网站。网站需要同时满足三个要求：

1. 结构清晰，能支撑长期学习而不是一次性阅读。
2. 内容足够完整，不只是目录，而是带详细模块讲解。
3. 实现足够轻量，本地可用，不依赖后端。

## 2. 目标与非目标

### 2.1 目标

本期交付的网站需要满足以下目标：

1. 将课程拆解为首页、阶段页、模块详情页三层结构。
2. 以课程结构数据驱动页面生成，避免手写大量静态页面。
3. 为每个模块提供详细学习内容，而不只是纲要标题。
4. 支持模块级完成勾选，并在本地浏览器中持久化。
5. 保持页面轻量、可离线浏览、可直接本地运行。
6. 视觉上体现“AI 架构师训练路线”的学习气质，而不是通用博客模板。

### 2.2 非目标

以下内容明确不在第一版范围内：

1. 用户账号、登录、云端同步。
2. 后端服务、数据库、管理后台。
3. 站内全文搜索。
4. 评论、讨论区、笔记系统。
5. 在线代码沙箱、运行环境、自动判题。
6. 多人协作或学习分享能力。
7. 超出 `GUIDE.md` 主线的大规模额外课程体系扩张。

## 3. 用户与使用场景

### 3.1 目标用户

唯一目标用户是当前项目作者本人。该用户会在本地浏览器中长期使用网站进行自学，不要求跨设备同步，也不要求多人访问。

### 3.2 使用场景

网站需要覆盖三个核心场景：

1. 首次进入时快速理解整条学习路线。
2. 按阶段进入模块，持续阅读和推进。
3. 在读完某个模块后手动标记完成，并在后续访问中保留进度。

## 4. 方案选择

本项目采用 `Astro 静态站点 + 结构化课程数据 + Markdown 内容 + 少量原生客户端脚本`。

选择理由：

1. `Astro` 适合文档式多页面站点，生成结果轻量。
2. 静态生成足以覆盖当前需求，不需要引入服务器。
3. 内容和模板分离后，后续调整课程目录的维护成本更低。
4. 模块完成状态仅需浏览器端持久化，用少量脚本即可实现。

不采用纯 HTML 多页面的原因是维护成本过高；不采用 React 单页应用的原因是当前交互需求过少，整体复杂度不必要。

## 5. 课程范围

课程结构以 [GUIDE.md](/Users/wangxin/Documents/llm-learn/self-teacing-llm/GUIDE.md) 为主事实来源。`GUIDE.md` 负责定义阶段边界、模块编号、模块标题、学习顺序和阶段项目；实现中允许在结构数据层补充展示所需的辅助元数据，但这些补充字段不得改变 `GUIDE.md` 已定义的课程范围与顺序。

课程包含以下 6 个阶段：

1. AI 工程基础
2. RAG 核心原理
3. RAG 进阶优化
4. Agent 核心原理
5. Agent 进阶与编排
6. 生产级 AI 系统

模块数量共 42 个，按阶段编号组织：

1. 第一阶段：1.1 到 1.7
2. 第二阶段：2.1 到 2.7
3. 第三阶段：3.1 到 3.7
4. 第四阶段：4.1 到 4.6
5. 第五阶段：5.1 到 5.8
6. 第六阶段：6.1 到 6.7

每个阶段的“实战项目”需要在阶段页中作为独立内容块展示，并在模块页中通过关联导航提示其所属阶段项目。

## 6. 信息架构

网站采用三层信息架构，并额外提供一个模块索引页：

### 6.1 首页

首页职责是建立全局认知，不承担全部正文内容。

首页必须包含：

1. 网站定位区：课程标题、学习定位、核心口号。
2. 学习路线图区：6 个阶段的概览卡片。
3. 学习方法论区：展示 `概念讲解 -> 架构图解 -> 代码识谱 -> Spec 练习 -> 实战项目`。
4. 课程特色区：理论、实战、Spec Coding。
5. 继续学习入口：默认跳转到全局学习顺序中的第一个未完成模块；如果全部模块已完成，则跳转到模块索引页。

### 6.2 阶段页

每个阶段单独一页，承担章节入口职责。

阶段页必须包含：

1. 阶段标题与阶段目标。
2. 阶段导读，解释本阶段解决什么问题。
3. 模块列表，按学习顺序展示。
4. 每个模块的完成状态。
5. 本阶段实战项目说明。
6. 前后阶段导航。

### 6.3 模块详情页

模块详情页是网站的核心学习页。

每个模块页必须包含：

1. 模块标题、编号、所属阶段。
2. 模块导读，说明为什么学它。
3. 核心知识讲解，基于 `GUIDE.md` 的条目扩写。
4. 关键对比或常见误区。
5. 学习检查点。
6. 实践建议。
7. 标记完成按钮。
8. 上一个 / 下一个模块导航。

### 6.4 模块索引页

增加一个“全部模块”索引页，方便后续跳转与回顾。索引页按阶段分组列出所有模块，支持查看模块完成状态，但不承担正文阅读。

## 7. 内容模型

内容必须拆分为“结构数据”和“正文内容”两层，不能把课程内容直接硬编码在页面模板中。

### 7.1 结构数据

结构数据负责描述课程层级和导航关系。阶段与模块的 `number`、`title`、`stageId`、顺序关系必须与 `GUIDE.md` 保持一致；其余辅助字段允许由网站内容作者手工补充。

每个阶段至少包含以下字段：

1. `id`
2. `number`
3. `title`
4. `slug`
5. `goal`
6. `summary`
7. `project`
8. `moduleIds`

每个模块至少包含以下字段：

1. `id`
2. `number`
3. `title`
4. `slug`
5. `stageId`
6. `summary`
7. `estimatedTime`
8. `prerequisites`
9. `keyPoints`

其中：

1. `number`、`title`、`stageId`、课程顺序来自 `GUIDE.md`。
2. `summary`、`estimatedTime`、`prerequisites`、`keyPoints` 为网站展示服务，允许人工编写并保存在网站的结构化数据文件中。
3. 人工补充字段只能用于增强学习体验，不能新增 `GUIDE.md` 中不存在的课程模块，也不能改变原有模块顺序。

### 7.2 正文内容

每个模块正文单独维护，正文以结构化 Markdown 为主，至少需要支持以下版块：

1. 为什么学这个
2. 核心知识
3. 关键对比 / 易错点
4. 学习检查点
5. 实践建议

正文内容需要以 `GUIDE.md` 为基础展开，但允许为了教学清晰度进行改写、补充解释、增加对比示例和学习提示。正文目标是“足够支持自学理解”，而不是百科式扩展。

## 8. 页面与组件设计

### 8.1 全局布局

全站采用统一的文档式布局，但避免直接复刻常见默认文档站模板。

布局要求：

1. 顶部保留轻量导航。
2. 桌面端使用左侧目录导航 + 右侧正文主区。
3. 移动端目录折叠到抽屉或折叠菜单。
4. 模块页正文阅读宽度需要受控，优先保证阅读体验。

### 8.2 核心组件

第一版至少需要以下组件：

1. 顶部导航栏
2. 路线图阶段卡片
3. 阶段模块列表
4. 模块完成按钮
5. 进度状态徽标
6. 上一个 / 下一个模块导航
7. 内容区块组件
8. 阶段项目说明卡片

组件职责要单一。课程内容展示组件不负责状态存储；状态组件不负责决定课程结构。

## 9. 交互设计

第一版仅实现必要交互。

### 9.1 必做交互

1. 模块完成 / 取消完成。
2. 当前模块在目录中的高亮。
3. 首页展示继续学习入口，规则固定为“跳到第一个未完成模块”。
4. 阶段页与索引页同步显示模块完成状态。
5. 模块页的前后模块跳转。

### 9.2 不做交互

第一版不实现：

1. 搜索框与全文过滤。
2. 收藏或稍后学习。
3. 自动记录阅读位置。
4. 评论、评分、笔记。
5. 复杂图表交互或在线实验面板。

## 10. 进度记录与数据持久化

进度记录只保存在浏览器本地 `localStorage`。

### 10.1 数据要求

数据至少需要表达：

1. 模块唯一标识。
2. 模块完成状态。

推荐采用单个命名空间键存储，例如：

`self-teaching-llm-progress-v1`

值结构为模块 `id -> boolean` 的映射。

继续学习入口不得依赖“最近访问位置”或额外阅读轨迹数据，而是根据全局模块顺序，从前到后计算第一个未完成模块作为目标页。

### 10.2 行为要求

1. 页面加载时读取本地进度并刷新 UI。
2. 用户点击完成按钮后立即更新界面与存储。
3. 在阶段页、索引页、模块页之间切换时状态保持一致。
4. 如果浏览器不支持或禁用了 `localStorage`，网站仍可阅读，但需要清晰提示“无法保存学习进度”，且继续学习入口退化为跳转到第一个模块。

## 11. 数据流

整体数据流必须保持简单清晰：

1. 构建期读取课程结构数据和模块正文内容。
2. Astro 根据数据生成首页、阶段页、模块页、索引页。
3. 运行期页面脚本读取本地进度状态。
4. 进度状态影响模块按钮、列表展示和继续学习入口目标。
5. 用户变更状态后写回浏览器本地存储。

运行期不得依赖远程接口，也不引入任何服务端状态。

## 12. 视觉方向

网站整体风格定位为“AI 架构师训练路线”，而不是普通博客或产品营销页。

视觉方向要求：

1. 首页强调路线图和能力进阶。
2. 阶段页强调训练感和任务感。
3. 模块页以可读性优先，减少装饰干扰。
4. 使用统一设计变量管理颜色、间距、字号与圆角。
5. 配色应冷静、专业、略带实验室感，但不能压暗到影响阅读。
6. 桌面与移动端都必须完整可用。

## 13. 内容写作标准

模块正文内容需要遵守以下写作标准：

1. 面向工程理解，不写成学院派教材。
2. 强调“为什么”和“如何判断”，不堆术语定义。
3. 多用概念对比和常见误区帮助记忆。
4. 内容密度适中，能完整读完，不造成过度压迫。
5. 语言风格统一，避免不同模块像不同来源拼接而成。

## 14. 错误处理与降级策略

第一版必须明确处理以下异常场景：

1. 模块正文缺失：页面展示清晰的缺失提示，而不是空白页。
2. 结构数据缺字段：在构建阶段显式报错。
3. `localStorage` 不可用：页面继续可读，但禁用保存进度并告知用户。
4. 无 JavaScript：网站仍能浏览课程正文，只是不能保存或展示动态进度状态；首页“继续学习入口”在此场景下退化为跳转到模块索引页。

## 15. 测试与验证要求

虽然第一版是轻量网站，但仍需覆盖基础验证。

### 15.1 构建验证

1. 所有页面都能成功生成。
2. 所有阶段页、模块页、索引页链接有效。
3. 内容数据与正文内容可以完整关联。

### 15.2 交互验证

1. 模块完成 / 取消完成逻辑正确。
2. 刷新页面后状态仍能保留。
3. 阶段页与索引页状态展示一致。
4. 继续学习入口不会跳到不存在的页面。

### 15.3 响应式验证

1. 首页、阶段页、模块页在桌面端可正常阅读。
2. 首页、阶段页、模块页在移动端可正常浏览和操作。

## 16. 第一版交付范围

第一版交付必须包含：

1. 一个可本地运行的静态课程网站工程。
2. 首页、全部阶段页、全部模块页、模块索引页。
3. 来自 `GUIDE.md` 的完整课程结构映射。
4. 为全部模块提供详细正文内容。
5. 模块级进度记录。
6. 统一的视觉样式与导航体验。

这里的“可本地运行”定义为：通过 Astro 开发服务器、预览服务器或任意本地静态 HTTP 服务访问即可；第一版不要求构建产物在 `file://` 直接打开时仍具备完整功能。

## 17. 验收标准

以下条件同时满足时，第一版视为达标：

1. 用户可以从首页清楚理解 6 个阶段的课程结构。
2. 用户可以进入任一阶段，再进入任一模块进行连续学习。
3. 每个模块页都不只是纲要，而有完整学习内容。
4. 用户可以对任一模块进行完成勾选，并在刷新后保留状态。
5. 网站在不依赖后端、不登录、不联网的情况下可正常使用。
6. 用户无需翻阅原始 `GUIDE.md`，即可在网站内完成路线理解、阶段跳转、模块阅读和完成标记这四类核心学习动作。

## 18. 实现约束

实现阶段需要遵守以下约束：

1. 轻量优先，不为少量交互引入重型客户端架构。
2. 内容和模板必须分离。
3. 不提前实现搜索、账户或云同步等第二阶段能力。
4. 页面设计要服务于学习效率，而不是展示技术堆栈。

## 19. 后续规划入口

本设计文档只定义第一版学习网站。若第一版完成并稳定使用，后续可以单独立项讨论以下扩展：

1. 搜索
2. 收藏 / 稍后学习
3. 自动记录阅读位置
4. 进度导出与导入
5. 更丰富的实验区与示意图

这些能力不属于本次实现计划的一部分。
